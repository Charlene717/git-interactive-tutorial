STEP 5 / 9

分支 (Branching)

如果 commit 是時間點，那麼 branch 就是平行時空——而且在 Git 裡，開一條分支只是改一個 41 bytes 的檔案，幾乎免費。

If commits are timepoints, branches are timelines. In Git a new branch is just a 41-byte file change — essentially free.

概覽

什麼是 Git 分支？

在 Git 裡，branch（分支）就是一個指向 commit 的輕量指標——它不是檔案的副本，也不會複製整個專案。所以建立、切換、刪除分支都幾乎瞬間完成（微秒等級），對效能毫無影響。

如果我們把每個 commit 想成歷史時間軸上的一個「時間點」，那麼分支就是這條時間軸的「平行時空」：你可以從同一個起點分出無限多條獨立路線，每條都能各自向前累積 commit，互不干擾。等到實驗成功，就把分支合回主線；失敗了，整條丟掉，主線完全不受影響。

In Git, a branch is just a lightweight pointer to a commit — not a copy of files, not a duplicate of the project. Creating, switching, and deleting branches are essentially free operations (microseconds), with no performance cost.

If a commit is a "moment in time" on a historical timeline, a branch is a parallel timeline: from any starting point you can spawn unlimited independent lines, each accumulating its own commits without interference. If the experiment works, merge it back; if not, throw the whole branch away — the main line stays untouched.

💡
核心心法：Git 的分支是「免費」的。其他 VCS（如 SVN）開分支等於複製整個專案目錄，又慢又貴；Git 只是動一個指標，所以鼓勵你動不動就開分支——topic branch、實驗 branch、WIP branch，用完就丟。 Mental shift: Git branches are free. Other VCS systems (e.g., SVN) copy the entire project tree — slow and expensive. Git just moves a pointer, so you're encouraged to branch liberally — topic, experimental, WIP branches — and discard them freely.
心智模型

三個核心概念

🌿

Branch（分支）

就是一個可以移動的指標，指向某個 commit。當你在這條分支上做新的 commit，分支指標會自動往前移到最新的 commit——你不用手動更新。

A movable pointer to a commit. When you commit on a branch, the pointer automatically advances to the new commit — you don't need to update it manually.

🎯

HEAD

一個特殊指標，指向「目前所在分支」。它告訴 Git「下次 commit 要放到哪一條分支」。cat .git/HEAD 通常會看到 ref: refs/heads/main。

A special pointer to your current branch — it tells Git where the next commit will go. Running cat .git/HEAD usually shows ref: refs/heads/main.

📌

Detached HEAD（游離狀態）

當 HEAD 直接指向某個 commit（而不是分支）時就進入「游離 HEAD」。git checkout <sha> 會觸發；Git 會警告：在這狀態下做 commit，若沒開新分支，這些 commit 容易變成孤兒，被 garbage collect 回收。

HEAD points directly at a commit (not a branch). Triggered by git checkout <sha>. Git warns you: commits made here are orphans unless you create a branch — eventually garbage-collected.

常用指令

分支的日常操作

Git 2.23+ 引入了 git switch 與 git restore，把原本被 git checkout 一個指令身兼多職的功能拆開。新寫法語意更清楚，這裡優先示範。

Git 2.23+ introduced git switch and git restore, splitting what git checkout used to overload. The new syntax is clearer — we use it first.

# 列出 / 檢視 / List & inspect
$ git branch                  # 列出本機分支 / list local
$ git branch -a               # 含遠端 / incl. remote
$ git branch -v               # 顯示每分支最後一個 commit / last commit per branch
$ git branch --merged         # 已合併到目前分支 / merged into HEAD
$ git branch --no-merged      # 尚未合併 / not yet merged

# 建立 / Create
$ git branch feature-x        # 只建立指標，不切過去 / create only
$ git switch -c feature-x     # 建立並切過去（推薦）/ create + switch
$ git checkout -b feature-x   # 舊寫法（仍然能用）/ legacy syntax

# 切換 / Switch
$ git switch main             # 現代 / modern
$ git checkout main           # 舊寫法 / legacy

# 重新命名 / Rename
$ git branch -m old-name new-name

# 刪除 / Delete
$ git branch -d feature-x     # 安全刪除：若未合併會拒絕 / safe — refuses if unmerged
$ git branch -D feature-x     # 強制刪除：可能遺失 commit / force, lossy
✨
為什麼推薦 switch？checkout 在歷史上被塞了太多功能（切分支、還原檔案、進 detached HEAD…），對新手非常混亂。switch 只做「切換分支」一件事，語意清楚。 Why prefer switch? checkout historically did too many things (switch branches, restore files, enter detached HEAD…) — confusing for newcomers. switch does one thing: switch branches.
觀念對比

為什麼 Git 的分支「不一樣」？

如果你之前用過 SVN、Perforce 或其他集中式 VCS，會記得「開分支」是大工程：要複製整個目錄樹、可能要先請示 lead，整個團隊都會看到。所以你不太敢隨便開分支。

Git 完全不同。一條分支只是 .git/refs/heads/<name> 底下的一個 41-byte 檔案（裡面只放一個 SHA-1）。沒有複製、沒有交易、沒有人會看到——直到你 push 上去之前，這條分支只存在你的本機。

結果是：Git 使用者開分支非常隨意。修一個小 bug 開一條、試一個新演算法開一條、寫一半的 WIP 開一條……完成後合回去，或直接 git branch -D 丟掉。這種「拋棄式分支（throwaway branch）」文化，是 Git 跟其他 VCS 最大的差別之一。

If you've used SVN, Perforce, or another centralized VCS, "branching" was a big deal: copy the whole tree, sometimes ask the lead, and the whole team sees it. So you rarely branched.

Git is completely different. A branch is a 41-byte file under .git/refs/heads/<name> (just one SHA-1). No copy, no transaction, nobody else sees it — until you push, the branch is only on your machine.

The result: Git users branch freely. A tiny bug fix? Branch. A new algorithm to try? Branch. A WIP idea? Branch. Merge back when done, or git branch -D to discard. This "throwaway branch" culture is one of Git's biggest cultural differences from older VCS.

決策流程

什麼時候該開新分支？

🌳 新分支決策樹

Q1:
正在嘗試一個有風險的想法？→ 是 → 開分支。失敗就整條丟掉，主線完全沒事。
Q2:
這個 feature 預估要超過 1 小時才能寫完？→ 是 → 開分支，避免半成品卡在 main。
Q3:
只是自己一個人在用的腳本，修一個 typo？→ 視情況 → 直接 commit 到 main 也行。
Q4:
多人同時改不同部分？→ 一定要 → 一個 feature 一條 branch，後面才能用 PR review。
Q5:
準備投稿論文要凍結程式碼？→ 是 → 開 release/manuscript-v1，之後 main 可以繼續開發、不影響快照。
Q1:
Trying out a risky idea? → Yes → Branch. If it fails, throw the branch away — main is untouched.
Q2:
Working on a feature that takes more than ~1 hour? → Yes → Branch — don't leave half-done work on main.
Q3:
Fixing a tiny typo on a script you maintain alone? → Optional → A direct commit to main is fine.
Q4:
Multiple people working on different things? → Definitely → One branch per feature, so PR review is possible.
Q5:
Preparing for paper submission? → Yes → Branch release/manuscript-v1 so further dev continues on main without affecting the snapshot.
完整範例

分支生命週期完整範例

# 從 main 出發、建立並切到 feature 分支 / Create + switch
$ git switch main
$ git switch -c feature/add-volcano-plot

# 編輯檔案後 commit / Edit + commit
$ echo "plot_volcano()" >> analysis.R
$ git add analysis.R
$ git commit -m "feat: add volcano plot"

# 切回 main / Switch back
$ git switch main

# 確認 feature 已合併後安全刪除（合併示範見 Step 6） / Delete
$ git branch -d feature/add-volcano-plot

# 舊版寫法：checkout -b 一次完成 create + switch / Legacy
$ git checkout main
$ git checkout -b feature/add-volcano-plot

# 編輯後 commit / Edit + commit
$ echo "plot_volcano()" >> analysis.R
$ git add analysis.R
$ git commit -m "feat: add volcano plot"

# 切回 main / Switch back
$ git checkout main

# 刪除 / Delete
$ git branch -d feature/add-volcano-plot

# 場景：你維護 scrna-pipeline，要加入 sctransform 正規化 step
# Scenario: maintain scrna-pipeline, add sctransform normalization

$ cd ~/projects/scrna-pipeline
$ git switch main
$ git pull origin main                     # 先同步遠端 / sync remote
$ git switch -c feature/sctransform

# commit 1: 加入函式骨架 / scaffold function
$ vim R/normalize.R
$ git add R/normalize.R
$ git commit -m "feat(normalize): add sctransform skeleton"

# commit 2: 整合到主 pipeline / wire into pipeline
$ vim R/run_pipeline.R
$ git commit -am "feat(pipeline): wire sctransform into run_pipeline"

# commit 3: 加單元測試 / add unit tests
$ vim tests/testthat/test-normalize.R
$ git add tests/testthat/test-normalize.R
$ git commit -m "test(normalize): cover sctransform path"

# 確認分支狀態，準備 merge（合併方式見 Step 6） / Review & prep merge
$ git log --oneline main..HEAD             # 看 feature 比 main 多了哪些 commit
$ git switch main
$ git branch --no-merged                   # feature/sctransform 應出現於此
# 接下來：Step 6 會教 git merge / git rebase
🧬
生資場景小提示：讓 main 永遠保持「目前已發表 / 已投稿」的版本；任何新分析開 feature/... 分支跑，跑通、review 過再合進來。這樣論文 reviewer 來信要復現結果時，git checkout main 就是當初提交的版本。 Bioinformatics tip: keep main as the "currently published / submitted" version; do new analyses on feature/... branches and merge only after review. When a reviewer asks you to reproduce a result, git checkout main is exactly the submitted version.
⚠️
共用專案守則：千萬不要直接 commit 到 main。一定要先開 feature 分支、push 上去開 Pull Request（Step 8），讓組員 review。直接推到共用 main 等於跳過所有把關，是最常見的協作事故來源。 Shared-project rule: never commit directly to main. Always branch + push + open a Pull Request (Step 8) so teammates can review. Direct commits to a shared main bypass all safeguards and are the single most common source of collaboration accidents.
