儲存庫基礎 (Repository Basics)

Git 最核心的心智模型——工作區、暫存區、儲存庫——以及四個每天都會用到的指令：status、add、commit、log。本章全程離線，還用不到 GitHub。

The Git mental model — working directory, staging area, repository — plus the four commands you'll use every day: status, add, commit, log. Everything here is offline; GitHub isn't needed yet.

概覽

為什麼需要 repo？

一個 repository (repo) 就是一個被 Git 追蹤的資料夾——裡面有你的程式、腳本、報告，還有一個隱藏的 .git/ 子目錄，存放所有歷史紀錄。每次你 commit，Git 就把當下整個專案的 snapshot 壓縮存到 .git/，並用一個 SHA-1 hash 當作這次 snapshot 的「身分證」。

本機 (local) vs 遠端 (remote)：本章只討論本機 repo——你的電腦上的資料夾。整個 Step 2 不需要連網，也不需要 GitHub 帳號。等到 Step 7（Remotes）我們才會把本機 repo 推上 GitHub。先把本機流程練熟，遠端只是「再多一個 push」。

A repository (repo) is simply a folder that Git tracks — your code, scripts, and reports, plus a hidden .git/ subdirectory that stores all history. Every commit packs the whole project's snapshot into .git/ and identifies it with a SHA-1 hash that acts as the snapshot's "ID card."

Local vs remote: this chapter is entirely about local repos — folders on your own machine. You don't need internet or a GitHub account for any of Step 2. We'll push to GitHub in Step 7 (Remotes). Master the local workflow first; the remote is just "one extra push."

💡
核心原則：Git 的全部魔法都在 .git/ 這個資料夾。刪掉它，這個資料夾就退回一般檔案夾；複製整個專案（含 .git/）到隨身碟，就帶走了完整的版本歷史。 Core principle: all of Git's magic lives in .git/. Delete it and the folder reverts to a plain directory; copy the whole project (including .git/) to a USB stick and you've taken the entire history with you.
三大區域

一、Git 三大區域：工作區 / 暫存區 / 儲存庫

這是 Git 最關鍵的心智模型——所有疑難雜症都從「我的檔案到底在哪一區？」開始。一個檔案的修改要進入版本歷史，必須走過這三區：

This is the most critical mental model in Git — every troubleshooting question starts with "which area is this file in?" To enter the version history, a change must travel through three areas:

📂

1. 工作區 (Working Directory)

就是你資料夾裡看得到、編輯器打得開的檔案。改了 analysis.R 卻還沒做任何 git 指令時，它就只在工作區。Git 知道有變動（因為它有上一次 commit 的版本可比），但還沒準備保存。

Just the files you see in the folder and open in your editor. If you edit analysis.R but haven't run any git command, the change lives only in the working directory. Git notices the change (it has the last commit to compare against) but hasn't been told to save it.

📥

2. 暫存區 (Staging / Index)

git add <file> 會把改動移到暫存區。可以把暫存區想成「下一次 commit 的購物車」——你挑幾個檔案丟進去，準備一起結帳。同一次工作可以做很多次 add，把不同的邏輯改動分成數個 commit。

git add <file> moves the change into the staging area. Think of it as a shopping cart for the next commit — you pick which files go in, then check out. You can add multiple times in one session, splitting different logical changes into separate commits.

🗄️

3. 儲存庫 (Repository / .git/)

git commit -m "..." 把購物車內所有東西結帳——做出一個新的 snapshot 永久存進 .git/，並產生一個 SHA hash 當作這個 commit 的 ID。commit 進去後就不會輕易消失（Git 的時光機能力靠的就是這個）。

git commit -m "..." checks the cart out — creates a new snapshot permanently stored in .git/, identified by a SHA hash. Once committed, the snapshot almost never disappears (this is what makes Git's time-travel possible).

# 三區流程 / The 3-area flow
   [Working Dir]  ──── git add ────▶  [Staging]  ──── git commit ────▶  [Repository]
   edit FASTQ_qc.R       stage selected hunks      snapshot saved (SHA: a3f9c1...)
建立 repo

二、兩種建立 repo 的方式：init vs clone

你只會碰到兩種情境：（A）我已經有一堆檔案，想開始追蹤——用 init；（B）我要拿別人雲端上的 repo——用 clone。

You'll only ever face two scenarios: (A) I already have files and want to start tracking them — use init; (B) I want a copy of someone's cloud repo — use clone.

# 在既有資料夾啟動版本控制 / Start tracking an existing folder
$ cd ~/projects/scRNA-paper
$ git init
Initialized empty Git repository in /home/charlene/projects/scRNA-paper/.git/

# 看看裡面有什麼 / Peek inside
$ ls -la
.git/   analysis.R   data/   README.md

# 此時 git status 會顯示所有檔案都是 "untracked"
$ git status
On branch main
No commits yet
Untracked files:
  analysis.R
  data/
  README.md
# 從 GitHub 取得完整 repo（含所有歷史）/ Get a full copy from GitHub
$ git clone git@github.com:satijalab/seurat.git
Cloning into 'seurat'...
remote: Counting objects: 12453, done.
Receiving objects: 100% (12453/12453), 28.4 MiB | 5.21 MiB/s, done.

# 進去看看 / Enter the folder
$ cd seurat
$ git log --oneline | head -3
3a8f2c1 (HEAD -> main, origin/main) v5.0.1 release
b7e91d4 fix: handle empty cell barcodes
f12a3b8 docs: update README
# Shallow clone — 只抓最新 1 個 commit（省時、省空間） / Latest snapshot only
$ git clone --depth 1 git@github.com:nf-core/rnaseq.git
# 適合：只是要跑 pipeline，不需要歷史。檔案小 60%~95%

# 帶子模組一起 clone / Include submodules
$ git clone --recurse-submodules git@github.com:lab/pipeline.git
# 適合：repo 用 git submodule 把工具當依賴，例如包了 Seurat 特定版本

# 只 clone 單一分支（譬如 dev）/ Clone a single branch
$ git clone --branch dev --single-branch git@github.com:lab/project.git
# 適合：分支極多的 repo，只關心 dev 那一條
📝
命名建議：repo 資料夾名稱建議用 kebab-case（如 scrna-aging-paper），避免空格、大寫、和中文字——跨平台、跨終端都不會出狀況。 Naming tip: use kebab-case for repo folder names (e.g., scrna-aging-paper) — no spaces, no uppercase, no non-ASCII. It travels safely across platforms and shells.
日常四大指令

三、每天必用：status / add / commit / log

🔎 git status

每次操作前先打它——告訴你「現在我的三區長什麼樣」。看懂這個輸出，Git 就懂一半了。

Run before any operation — it tells you "what does my 3-area state look like right now?" Master this output and you've learned half of Git.

$ git status
On branch main
Changes to be committed:    # 已 add，在暫存區
  modified: scripts/qc.R

Changes not staged for commit:  # 改了但沒 add
  modified: scripts/de_analysis.R

Untracked files:            # Git 沒看過的新檔
  results/figure1.pdf

➕ git add

把改動丟進暫存區。三種常見用法：

Move changes into staging. Three common forms:

# 加單一檔 / single file
$ git add scripts/qc.R

# 加全部變動 / everything in CWD
$ git add .

# Patch 模式：互動式挑要不要 add 每段 hunk
$ git add -p scripts/qc.R
# 適合：一個檔案有多種邏輯改動，想分開 commit

💾 git commit

把暫存區結帳，產生一個 snapshot。-m 直接給 message；不給的話會打開編輯器。

Check out the staging cart, producing a snapshot. -m supplies the message inline; without it your editor opens.

# 標準寫法 / standard
$ git commit -m "qc: drop cells with <200 genes"

# 一次 stage 所有已追蹤改動 + commit
# （注意：不會加新檔案 untracked files）
$ git commit -am "refactor de pipeline"

# Amend：修改最後一個 commit（補檔或改 message）
$ git add forgotten.R
$ git commit --amend --no-edit

📜 git log

查看歷史 commit。預設輸出很長，加旗標讓它變實用：

View commit history. The default is verbose — these flags make it useful:

# 一行一個 commit / one line per commit
$ git log --oneline
a3f9c1f (HEAD -> main) qc: drop low-quality cells
b7e91d4 init: import paper repo

# 看分支圖 / visualize branches
$ git log --oneline --graph --all --decorate

# 篩特定作者 / 特定路徑
$ git log --author="Charlene" -- scripts/qc.R
✍️
好的 commit message 慣例：第一行 ≤ 50 字，動詞開頭（add, fix, refactor, docs），描述「這個 commit 做了什麼」而非「我做了什麼」。例：fix: handle empty FASTQ files in qc step。 Good commit-message convention: first line ≤ 50 chars, lead with a verb (add, fix, refactor, docs), describe what the commit does, not what you did. Example: fix: handle empty FASTQ files in qc step.

五、init 還是 clone？決策樹

🌳 開啟 repo 流程圖

Q1:
已經有 GitHub 上的 repo，想抓下來用？→ 是 → git clone <url>（含完整歷史 + 自動設好 origin remote）。
Q2:
本機只有一堆檔案，還沒上雲端？→ 是 → git init，做幾個 commit，之後想 push 時再去 GitHub 建空 repo + git remote add origin。
Q3:
要加入別人的開源專案開發？→ 是 → 先在 GitHub 介面 fork 到自己帳號，再 git clone 你的 fork（不是原專案）。
Q4:
要 clone 但只要最新版（譬如下載 nf-core pipeline 跑一次）？→ 是 → git clone --depth 1，可以省下大量磁碟和時間。
Q1:
Want a copy of an existing GitHub repo? → Yes → git clone <url> (gets full history and auto-configures the origin remote).
Q2:
Local-only files, not in the cloud yet? → Yes → git init, make a few commits, then later create an empty GitHub repo and git remote add origin.
Q3:
Contributing to someone else's open-source project? → Yes → Fork on GitHub first, then git clone your fork (not the original).
Q4:
Cloning just to run the latest version (e.g., downloading an nf-core pipeline)? → Yes → git clone --depth 1 — saves a lot of disk and time.

六、典型第一個 commit 範例

# 全新本機專案：從零到第一個 commit / From nothing to first commit
$ mkdir ~/projects/my-thesis
$ cd ~/projects/my-thesis

# 啟動 Git / initialize
$ git init
Initialized empty Git repository in /home/charlene/projects/my-thesis/.git/

# 建幾個檔案 / create some files
$ echo "# My Thesis" > README.md
$ echo "library(Seurat)" > analysis.R

# 看 Git 怎麼看 / inspect
$ git status   # 兩個檔案都是 untracked

# 一次加入並 commit / stage all + commit
$ git add .
$ git commit -m "init: project skeleton"
[main (root-commit) a3f9c1f] init: project skeleton
 2 files changed, 2 insertions(+)
# 抓論文程式碼下來研究 / Pull a paper's code to study
$ git clone https://github.com/satijalab/azimuth.git
Cloning into 'azimuth'...

$ cd azimuth

# 看一下歷史結構 / poke at history
$ git log --oneline | head -5
$ git branch -a              # 列出所有分支

# 建一個自己的 working 分支，避免動到 main
$ git switch -c my-experiments
# （Step 5 會詳談分支——這裡先讓你看到 clone 之後常見的下一步）
# 真實情境：開始一個 scRNA-seq 分析專案 / Start a scRNA-seq project
$ mkdir ~/projects/scrna-aging-2026
$ cd ~/projects/scrna-aging-2026

$ git init
$ git branch -M main          # 確保預設分支叫 main

# 標準資料夾結構 / standard layout
$ mkdir -p data scripts results docs
$ echo "# scRNA Aging 2026" > README.md
$ touch scripts/01_qc.R scripts/02_clustering.R

# 重要：data/ 是大檔，不該被 commit
# 先建 .gitignore（Step 9 會詳細討論）
$ cat > .gitignore << EOF
data/
results/*.h5ad
results/*.pdf
.Rhistory
.RData
EOF

$ git status               # data/ 不再出現了！
$ git add .
$ git commit -m "init: project skeleton with .gitignore"
地雷區

七、常見地雷 ⚠️

⚠️
不要在另一個 git repo 內 git init！會產生巢狀 .git/，外層 repo 完全看不到內層的歷史，未來 push 會崩。先 git status 確認自己不在某個既有 repo 裡（會印 fatal: not a git repository 才安全 init）。 Never git init inside another git repo! You'll create a nested .git/; the outer repo can't see the inner history, and pushes will break. Run git status first — if it prints fatal: not a git repository, you're safe to init.
🚨
不要刪 .git/ 資料夾！從 clone 來的 repo 裡的 .git/ 含有所有 commit、所有分支、所有 tag——刪了就退回成普通檔案夾，本機歷史完全消失（除非雲端還有）。看到 .git/ 又大又雜，請忍住不要清理。 Never delete .git/! The .git/ in a cloned repo holds every commit, every branch, every tag — delete it and the folder reverts to a plain directory with all local history gone (unless the cloud copy survives). If .git/ looks bulky and messy, resist the urge to tidy it.
📌
git commit -am 不會加新檔案！它只把已追蹤檔案的修改一次 stage + commit。新建的檔案還是要先 git add。 git commit -am doesn't add new files! It only stages and commits modifications to already-tracked files. New files still need git add first.
