STEP 1 / 9

安裝與設定 (Setup & Config)

把 Git 裝好、把身份設好、再選對認證方式——之後三年都不用回頭重做這一步。

Install Git, configure your identity, and pick the right auth method — do it once, and you won't have to revisit this step for three years.

概覽

為什麼要先做 Setup？

Git 是一個追蹤檔案變化的分散式版本控制系統；GitHub 則是把這些 repo 放上雲端、讓多人協作的平台。在開始任何分析專案之前，先做好三件事：(1) 確認本機有 Git；(2) 設定 commit 用的身分（姓名、email）；(3) 設好和 GitHub 之間的認證金鑰（SSH key 或 PAT）。

為什麼重要？沒設身分，每次 commit 都會用機器預設值（常常是 user@hostname），論文發表後想引用 commit 時會很尷尬。沒設 SSH/PAT，每次 push 都要打密碼；2021 年 8 月後 GitHub 已停止接受密碼認證，必須使用 SSH 或 Personal Access Token (PAT)。

Git is a distributed version-control system that tracks file history; GitHub is a cloud platform that hosts those repos for collaboration. Before any analysis project, do three things: (1) make sure Git is installed locally; (2) set your commit identity (name + email); (3) configure how you authenticate to GitHub (SSH key or PAT).

Why this matters: with no identity, every commit uses the system default (often user@hostname), which is embarrassing when a paper cites your commits. With no SSH/PAT, you'll be prompted for a password every push — and since August 2021, GitHub no longer accepts passwords, so you must use SSH or a Personal Access Token (PAT).

💡
核心原則：Git 設定一次走天涯。整體配置存在 ~/.gitconfig（全域）；每個 repo 內也可以有 .git/config 覆寫設定（例如用實驗室 email 而不是個人 email）。 Core principle: Configure Git once, use forever. Global settings live in ~/.gitconfig; each repo's .git/config can override them (e.g., use your lab email instead of your personal one).
安裝

一、依平台安裝 Git

🪟

Windows

到 git-scm.com 下載 Git for Windows（含 Git Bash、MinTTY、Git GUI）。安裝時建議：
• Default editor: VS Code
• Line endings: Checkout Windows, commit Unix
• Use bundled OpenSSH。

Download Git for Windows from git-scm.com (includes Git Bash, MinTTY, Git GUI). Recommended installer choices:
• Default editor: VS Code
• Line endings: Checkout Windows, commit Unix
• Use bundled OpenSSH.

🍎

macOS

三種方式：(1) xcode-select --install（系統內建）；(2) brew install git（最新版，推薦）；(3) 官方 .dmg。Apple Silicon 用 Homebrew 最新版較穩。

Three options: (1) xcode-select --install (system bundled); (2) brew install git (latest version, recommended); (3) official .dmg. On Apple Silicon, Homebrew's latest build is most reliable.

conda 環境：conda install -c conda-forge git（HPC 沒 root 時常用）。

Ubuntu/Debian: sudo apt install git
CentOS/RHEL: sudo dnf install git
conda env: conda install -c conda-forge git (handy on HPC where you have no root).

設定身份

二、設定 commit 身份

Git 把每次 commit 都標記為「誰、什麼時候、做了什麼」。姓名與 email 一定要設——email 還是 GitHub 對應作者頭像的關鍵。

Every Git commit is tagged with "who, when, what." You must set your name and email — the email is also how GitHub links commits to your profile avatar.

# 全域設定（所有 repo 共用） / Global (applies to all repos)
$ git config --global user.name "Charlene Yang"
$ git config --global user.email "charlene@lab.edu"

# 在單一 repo 內覆寫（例如：用實驗室 email） / Override per-repo
$ cd ~/projects/my-paper
$ git config user.email "charlene@university.edu"

# 查看現有設定 / View current config
$ git config --list --show-origin
$ git config user.email
🔒
隱私小技巧：GitHub 提供 noreply email（在 設定頁 找到 1234567+username@users.noreply.github.com），不想公開個人 email 時用它做 user.email 即可。 Privacy tip: GitHub offers a noreply email (find 1234567+username@users.noreply.github.com in your email settings). Use that as user.email if you don't want your personal email exposed in commits.
體驗優化

三、編輯器、預設分支與 alias

🖋️ 預設編輯器

# 用 VS Code / Use VS Code
$ git config --global core.editor "code --wait"

# 用 nano（簡單） / Use nano
$ git config --global core.editor nano

寫 commit message 或 rebase 時會打開。預設是 vi，新手常被卡住。

Used when writing commit messages or doing interactive rebases. Default is vi, which often traps newcomers.

🌱 預設分支名稱

# 把預設分支改為 main / Set default branch to main
$ git config --global init.defaultBranch main

GitHub 自 2020 年起預設用 main。本機若還是 master，會跟雲端不一致。

GitHub has used main by default since 2020. If your local default is still master, you'll diverge from cloud naming.

實用 alias 範例

# 把常用指令縮短 / Shorten common commands
$ git config --global alias.st status
$ git config --global alias.co checkout
$ git config --global alias.br branch
$ git config --global alias.lg "log --oneline --graph --decorate --all"
$ git config --global alias.last "log -1 HEAD"

# 之後就可以 / Now you can
$ git st        # = git status
$ git lg        # 漂亮的分支圖
認證

四、GitHub 認證：SSH 還是 PAT？

🌳 認證方式決策流程

Q1:
長期在同一台電腦工作？→ 是 → 用 SSH key（一勞永逸）。
Q2:
需要從 HPC / 共用伺服器存取，但不能放私鑰？→ 是 → 用 PAT（搭配 credential helper 暫存）。
Q3:
要在腳本 / CI 中自動 push？→ 是 → 用 Fine-grained PAT 或 GitHub Actions token，限制範圍與 repo。
Q4:
機構強制兩階段驗證？→ 是 → SSH 或 PAT 都需要——但仍能順利使用，不需要每次輸入 OTP。
Q1:
Working long-term on the same machine? → Yes → Use an SSH key (one-time setup).
Q2:
Need access from HPC / shared servers where you can't store a private key? → Yes → Use a PAT (with a credential helper for caching).
Q3:
Pushing from scripts / CI? → Yes → Use a fine-grained PAT or a GitHub Actions token scoped to specific repos.
Q4:
2FA enforced by your institution? → Yes → SSH or PAT both work — no need to type OTPs every time.

完整程式碼

完整設定範例

# 1. 設定身份 / Identity
git config --global user.name "Your Name"
git config --global user.email "you@example.com"
git config --global init.defaultBranch main
git config --global core.editor "code --wait"
git config --global pull.rebase false     # pull = merge (safer for newcomers)

# 2. 產生 SSH key (ed25519 為現代推薦) / Generate SSH key
ssh-keygen -t ed25519 -C "you@example.com"
# 一路按 enter，密碼可空 / Accept defaults

# 3. 啟動 ssh-agent 並加入 key
eval "$(ssh-agent -s)"
ssh-add ~/.ssh/id_ed25519

# 4. 複製公鑰 / Copy public key
# macOS:
pbcopy < ~/.ssh/id_ed25519.pub
# Linux:
xclip -sel clip < ~/.ssh/id_ed25519.pub
# Windows (Git Bash):
clip < ~/.ssh/id_ed25519.pub

# 5. 在 GitHub Settings → SSH keys 加入 → 測試
ssh -T git@github.com
# Hi USERNAME! You've successfully authenticated...
# 1. 設定身份（同上）/ Identity (same as SSH)
git config --global user.name "Your Name"
git config --global user.email "you@example.com"

# 2. 啟用 credential helper（讓 PAT 不用每次輸入）
#    macOS:
git config --global credential.helper osxkeychain
#    Windows:
git config --global credential.helper manager
#    Linux:
git config --global credential.helper "cache --timeout=86400"

# 3. 到 GitHub Settings → Developer settings → Personal access tokens → 
#    Fine-grained tokens → 點 Generate new token
#    勾選需要的 scopes: repo, workflow ...

# 4. 第一次 push 時，使用者名輸入 GitHub 帳號，密碼貼上 PAT
git push origin main
# Username: charlene717
# Password: ghp_xxxxxxxxxxxxxxxxxxxx
# （helper 會記住，下次不問）
# Windows 特有設定 / Windows-specific config

# 換行符號（避免 Linux/Windows 混亂）
git config --global core.autocrlf true

# 大小寫敏感（Windows 預設不敏感，跨平台時建議開）
git config --global core.ignorecase false

# 用 OpenSSH（內建在 Win 10+） / Use bundled OpenSSH
git config --global core.sshCommand "C:/Windows/System32/OpenSSH/ssh.exe"

# 中文檔名顯示正常 / Display Chinese filenames
git config --global core.quotepath false
⚠️
常見地雷：不要把 ~/.ssh/id_ed25519（私鑰）上傳到任何地方；只能上傳 ~/.ssh/id_ed25519.pub（公鑰）。Token 一旦外洩，立刻去 GitHub Settings 撤銷，否則任何人都能假冒你 push。 Common pitfalls: never upload ~/.ssh/id_ed25519 (the private key) anywhere — only ever the .pub file. If a token leaks, revoke it immediately in GitHub Settings; otherwise anyone can push as you.
生資現場

生物資訊實驗室常見場景

🖥️ 多台機器：個人 NB + HPC 帳號

每台機器都產自己的 SSH key（不要共享私鑰），全部加進 GitHub 帳號的 SSH keys 頁面。Label 加機器名稱（如 laptop-2026, HPC-login01），日後撤銷比較方便。

Generate a fresh SSH key on each machine (don't share private keys). Add all of them to your GitHub SSH keys page, with labels like laptop-2026 or HPC-login01 so you can revoke individually later.

🔐 HPC 沒法外連 GitHub port 22

在 ~/.ssh/config 加入：

Host github.com
  Hostname ssh.github.com
  Port 443
  User git

之後 ssh -T git@github.com 就會走 HTTPS port，多數 firewall 都放行。

Add to ~/.ssh/config:

Host github.com
  Hostname ssh.github.com
  Port 443
  User git

Then ssh -T git@github.com uses HTTPS port 443, which firewalls usually allow.

🧪 不同 repo 用不同 email

在每個 repo 內：git config user.email lab@uni.edu。或在 ~/.gitconfig 用 [includeIf "gitdir:~/work/"] 條件式 include 不同設定檔。

Per-repo: git config user.email lab@uni.edu. Or in ~/.gitconfig, use [includeIf "gitdir:~/work/"] to conditionally include a different config file.

🤝 實驗室帳號 vs 個人帳號

建議：個人 GitHub 留 portfolio；實驗室專案放到 lab organization（如 my-lab/scRNA-pipeline）。離職後 organization 仍保留 repo，所有人 commit 歷史完整。

Recommendation: keep your personal GitHub for portfolio work; put lab projects in a GitHub Organization (e.g., my-lab/scRNA-pipeline). When members leave, the org keeps the repo, and everyone's commit history stays intact.
