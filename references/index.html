📌 教學註記與細節

以下整理使用 Git / GitHub 工具時值得多注意的細節、政策異動與情境限制；部分屬於勘誤、多數屬於補充說明，協助讀者避免常見陷阱。

政策更動 · Policy change

預設分支名稱由 master 改為 main（2020-10）

GitHub 於 2020-10-01 起，所有新建 repository 預設分支改為 main。Git 自 2.28 起亦支援 init.defaultBranch 設定。教學中若仍看到 master，請以 git config --global init.defaultBranch main 統一。本地舊 repo 可用 git branch -m master main 並 git push -u origin main，再到 GitHub UI 將預設分支切換並刪除舊 master。

Since 2020-10-01, GitHub creates new repositories with main as the default branch. Git 2.28+ also supports init.defaultBranch. If tutorials still show master, standardize with git config --global init.defaultBranch main. For existing repos: git branch -m master main, then git push -u origin main, switch the default in the GitHub UI, and delete the old master.

來源：github/renaming · GitHub default branch rename · Software Freedom Conservancy 公告

Source: github/renaming · GitHub default branch rename · Software Freedom Conservancy announcement

指令分流 · Command split

git checkout 拆分為 git switch / git restore（Git 2.23, 2019-08）

Git 2.23 起新增兩個語意更清楚的指令：git switch <branch> 用於切換分支、git restore <file> 用於還原工作區檔案。原本 git checkout 同時承擔兩種職責，是 Git 新手最大混淆來源之一。Pro Git 第 2 版仍多以 checkout 示範，但在新教學中建議直接教 switch 與 restore。注意 switch -c 對應舊的 checkout -b。

Git 2.23 introduced two semantically cleaner commands: git switch <branch> for branch switching, and git restore <file> for restoring working-tree files. git checkout previously did both, which is a major source of confusion for newcomers. Pro Git 2e still uses checkout; new tutorials should prefer switch and restore. Note switch -c replaces the old checkout -b.

Source: git-switch official docs · GitHub Blog — Highlights from Git 2.23

安全強制 · Security mandate

GitHub 取消密碼驗證（2021-08-13）— 改用 PAT 或 SSH key

2021-08-13 起，GitHub 在 Git 操作（push/pull/clone over HTTPS）時不再接受帳號密碼驗證。必須改用 Personal Access Token (PAT)、SSH key 或 GitHub CLI (gh auth login) 配合 Git Credential Manager。Fine-grained PAT 自 2022 推出，可限制到特定 repo 與 scope，建議優先於 Classic PAT。若教學仍叫讀者輸入 GitHub 密碼，那是過時資訊。

Since 2021-08-13, GitHub no longer accepts account passwords for Git operations (push/pull/clone over HTTPS). Use a Personal Access Token (PAT), an SSH key, or GitHub CLI (gh auth login) with Git Credential Manager. Fine-grained PATs (since 2022) can be scoped to specific repos and permissions — prefer them over Classic PATs. Any tutorial still asking readers to type their GitHub password is outdated.

補充說明 · Notes

GitHub Actions secrets：Repository / Environment / Organization 三層

GitHub Actions 的 secrets 有三個層級：Repository（最基本）、Environment（可加 required reviewers 與 wait timer，2020 推出，2024 進一步擴充至 OIDC 條件式存取）、Organization（跨 repo 共用）。生資 pipeline 部署到 production HPC 時建議使用 Environment secrets + required reviewers，避免低權限 PR 直接觸發 production deploy。OIDC 機制可完全免去長期 secrets，建議搭配 AWS / GCP / Azure 使用。

GitHub Actions secrets have three scopes: Repository (basic), Environment (adds required reviewers and wait timers; introduced 2020, further expanded with OIDC conditional access in 2024), and Organization (shared across repos). For bioinformatics pipelines deploying to production HPC, use Environment secrets with required reviewers to prevent low-privilege PRs from triggering production deploys. OIDC eliminates long-lived secrets entirely — recommended with AWS / GCP / Azure.

額度限制 · Quota limit

Git LFS 免費額度：1 GB 儲存 + 1 GB/月頻寬

GitHub 的 Git LFS 免費額度極為有限：1 GB 儲存空間 + 1 GB/月下載頻寬。一個 BAM 檔（30× WGS 約 60–90 GB）就會立刻爆量。超量需購買 data pack（每包 $5/月，50 GB 儲存 + 50 GB 頻寬）。生資建議改用：Zenodo（開放 50 GB/dataset 免費 + DOI）、SRA / GEO（原始定序資料）、S3 / GCS（pipeline intermediates），repo 只放校驗碼與下載腳本。

GitHub's Git LFS free tier is very tight: 1 GB storage + 1 GB/month download bandwidth. A single 30× WGS BAM (~60–90 GB) will exceed it immediately. Overages require data packs ($5/month each for 50 GB storage + 50 GB bandwidth). Bioinformatics alternatives: Zenodo (free 50 GB/dataset + DOI), SRA / GEO (raw sequencing data), S3 / GCS (pipeline intermediates) — keep only checksums and download scripts in the repo.

工具補充 · Tool note

Jupyter notebook 提交前用 nbstripout 清除 outputs

.ipynb 檔案內嵌 base64 編碼的圖像 / 表格 outputs，會讓每次重跑 cell 都產生大量 diff，並可能洩漏中間資料。建議安裝 nbstripout 作為 git filter（nbstripout --install），自動在 staging 時剝除 outputs，pull 後本地仍可重跑。或改用 Jupytext 將 notebook 同步為 .py / .md（純文字 diff 友善，建議與 .ipynb 雙存）。

.ipynb files embed base64-encoded image/table outputs, producing huge diffs on every cell rerun and potentially leaking intermediate data. Install nbstripout as a git filter (nbstripout --install) to strip outputs automatically at staging time; outputs are regenerated locally on pull. Alternatively, use Jupytext to sync notebooks to .py / .md for diff-friendly text, ideally paired with .ipynb.

規範補充 · Spec note

Conventional Commits 不會被強制 — 需搭配 commitizen / commitlint

Conventional Commits 1.0.0（feat:, fix:, BREAKING CHANGE: 等）是規範，Git 本身不會驗證。要實質強制需：（1）pre-commit hook 跑 commitlint；（2）開發者用 commitizen 互動建立 commit；（3）CI 階段以 GitHub Action 阻擋不合規 PR。配合 semantic-release 可自動依 commit 類型決定 SemVer bump。

Conventional Commits 1.0.0 (feat:, fix:, BREAKING CHANGE:, etc.) is a spec — Git does not validate it. To enforce it: (1) run commitlint as a pre-commit hook; (2) use commitizen for interactive commit composition; (3) block non-compliant PRs with a CI GitHub Action. Combined with semantic-release, commit types can automatically drive SemVer bumps.

設計選擇 · Design choice

Submodule vs Subtree：常見混淆與如何選擇

Submodule 只記錄子 repo 的 commit pointer，clone 主 repo 不會自動拉子模組（需 git clone --recursive 或 git submodule update --init），新人最容易踩坑。Subtree 把子 repo 內容真的合進主 repo 歷史，clone 即完整。建議：需獨立發版且子 repo 也有自己團隊 → submodule；只是想引入第三方 reference data / 工具且偶爾更新 → subtree。生資 pipeline 內嵌的 reference genome / annotation 通常選 subtree（或外部 fetch）。

Submodule records only a commit pointer to the sub-repo; cloning the main repo does not pull submodules (needs git clone --recursive or git submodule update --init) — the top source of confusion for newcomers. Subtree merges the sub-repo's content into the main repo's history, so a clone is complete. Rule of thumb: separate release cycles with separate teams → submodule; vendoring third-party reference data / tools with occasional updates → subtree. Embedded reference genomes / annotations in bioinformatics pipelines usually use subtree (or external fetching).

版本漂移 · Version drift

ubuntu-latest runner 會自動升級 — 鎖版本避免「上週還好好的」

GitHub Actions 的 ubuntu-latest 標籤每隔數個月會自動指向新版（22.04 → 24.04 已於 2025-01-17 完成），預裝工具、Python 版本、GLIBC 都可能變動，常導致「上週還能跑、這週突然壞」。生資 reproducibility 建議：（1）workflow 鎖死版本（ubuntu-22.04）；（2）actions/checkout、setup-python 等也鎖 major（@v4）；（3）關鍵 release 用 commit SHA 鎖；（4）每年安排升級 sprint 一次性遷移。

GitHub Actions' ubuntu-latest label is rebound every few months (22.04 → 24.04 completed on 2025-01-17). Preinstalled tools, Python versions, and GLIBC can shift, causing silent breakage ("worked last week"). For reproducibility: (1) pin workflows to a fixed version (ubuntu-22.04); (2) pin action majors (actions/checkout@v4); (3) for critical releases, pin to commit SHAs; (4) schedule an annual upgrade sprint.

補充說明 · Notes

git pull --rebase 預設化：避免 merge bubble 污染歷史

預設 git pull 是 fetch + merge，多人同分支開發會產生大量「Merge branch 'main' of ...」噪音 commit。建議全域設定 git config --global pull.rebase true，pull 時自動 rebase 本地 commit 到遠端最新；遇到衝突再 git rebase --continue / --abort。Git 2.27+ 對未設定者會發出 hint 提示。注意：已 push 的 commit 不要 rebase（會改寫共享歷史）。

Default git pull is fetch + merge, which generates noisy "Merge branch 'main' of ..." commits in shared-branch development. Set git config --global pull.rebase true globally to rebase local commits onto the remote tip on pull; resolve conflicts with git rebase --continue / --abort. Git 2.27+ warns users who haven't configured this. Caveat: never rebase commits already pushed to a shared branch (it rewrites shared history).
