STEP 7 / 9

遠端與 GitHub 同步 (Remotes & GitHub Sync)

本機 repo 和 GitHub 上的 repo 之間，靠 fetch / pull / push 三個動詞溝通——理解它們，雲端協作就不再神秘。

Local repo and GitHub repo talk to each other via three verbs — fetch, pull, push. Once those click, cloud collaboration stops feeling magical.

概覽

什麼是 remote？

remote（遠端）本質上只是一個有名字的 URL，指向 repo 的另一份副本（通常在 GitHub 上）。大多數 repo 都有一個叫 origin 的 remote——可能是你自己的 fork，也可能是 lab 的 canonical repo。當你 git clone 時，Git 自動把來源設成 origin。

本章重點：本機與 remote 之間如何保持同步。所有同步動作都繞著三個動詞轉——git fetch（下載）、git pull（下載並合併）、git push（上傳）。再加上「tracking branch」這個概念，你就能掌握 GitHub 協作的核心機制。

A remote is just a named URL pointing to another copy of your repo (typically on GitHub). Most repos have a remote called origin — either your own fork, or the lab's canonical repo. When you git clone, Git automatically sets the source as origin.

This chapter is about how local and remote stay in sync. Everything revolves around three verbs — git fetch (download), git pull (download + merge), git push (upload). Add the idea of a "tracking branch," and you have the core mechanics of GitHub collaboration.

💡
核心原則：remote 只是別名。origin 不是 Git 內建的保留字，只是 git clone 預設取的名字。你可以隨時 git remote rename origin upstream 改名。 Core principle: a remote is just an alias. origin isn't a Git keyword — it's just the default name git clone assigns. You can git remote rename origin upstream any time.
管理 remote

一、管理 remote

新建一個本機 repo 時，要手動把它連到 GitHub。clone 下來的 repo 則自動有 origin。當你 fork 別人的 repo 後，慣例會把自己的 fork 設為 origin、原始 repo 設為 upstream。

For a freshly initialized local repo, you wire it to GitHub by hand. For a cloned repo, origin is already set. When you fork someone's repo, the convention is to keep your fork as origin and add the original repo as upstream.

# 列出所有 remote 與 URL / List remotes with URLs
$ git remote -v
origin  git@github.com:charlene/scRNA-pipeline.git (fetch)
origin  git@github.com:charlene/scRNA-pipeline.git (push)

# 新增 remote / Add a remote
$ git remote add origin git@github.com:charlene/scRNA-pipeline.git

# 切換 URL（例如 HTTPS → SSH） / Switch URL (HTTPS ↔ SSH)
$ git remote set-url origin git@github.com:charlene/scRNA-pipeline.git

# 改名 / Rename
$ git remote rename origin upstream

# 移除 / Remove
$ git remote remove old-mirror

# Fork workflow: 自己的 fork = origin, 原 repo = upstream
$ git remote add upstream git@github.com:lab/scRNA-pipeline.git
三個同步動詞

二、fetch / pull / push

📥

git fetch

把 remote 的 refs 和 objects 下載到本機，但不動工作目錄。本機多了 origin/main 之類的 remote-tracking ref，本地分支不變。最安全。

Downloads remote refs and objects to your machine, but doesn't touch your working tree. Updates remote-tracking refs like origin/main; your local branches stay put. Safest verb.

📥➕

git pull

等同 git fetch + git merge origin/<current-branch>（預設）。方便，但把兩個動作混在一起；衝突時新手容易亂。

Equivalent to git fetch + git merge origin/<current-branch> (default). Convenient but conflates two operations — newcomers get tangled when conflicts arise.

📤

git push

把本機 commit 上傳到 remote。第一次推新分支用 git push -u origin feature 設定 upstream，之後直接 git push 就懂去哪。

Uploads local commits to the remote. First time pushing a new branch, use git push -u origin feature to set the upstream — future git push calls know where to go.

# pull 的內部其實就是 fetch + merge / pull == fetch + merge
$ git pull
# 等價於 / Equivalent to:
$ git fetch origin
$ git merge origin/main

# 第一次推新分支 / First push of a new branch
$ git push -u origin feature-clustering
# 之後 / Later
$ git push
追蹤分支

三、tracking branch（追蹤分支）

本機的 main 通常會 track（追蹤）origin/main——這個 remote-tracking ref 是 fetch 後本機對 remote 狀態的快照。tracking 關係讓 git status 能說「你的分支領先 2 個 commit」或「落後 3 個 commit」。

沒有 tracking，git pull / git push 都要明寫 remote 和分支名。

Your local main typically tracks origin/main — that remote-tracking ref is a snapshot of remote state after the last fetch. The tracking relationship is what lets git status say "Your branch is ahead by 2 commits" or "behind by 3."

Without tracking, every git pull / git push needs explicit remote and branch names.

# 查看分支與其追蹤關係 / Show branches with tracking info
$ git branch -vv
* main          a3f12b9 [origin/main] Add QC step
  feature-umap  7d2e8c1 [origin/feature-umap: ahead 2] Tune n_neighbors

# 從遠端建立追蹤的本地分支 / Create a local branch tracking a remote one
$ git checkout --track origin/feature-clustering
# 簡寫 / Shortcut (Git ≥ 2.23):
$ git switch feature-clustering   # 若名稱不衝突會自動 --track

# 為現有分支設追蹤 / Set upstream for an existing branch
$ git branch -u origin/main main

四、fetch vs pull vs push

動詞 方向 動工作目錄？ 安全性
fetchremote → 本機 refs（不動分支）remote → local refs only否no最安全safest
pullremote → 本機 + 合併到當前分支remote → local + merge會yes大致安全（可能衝突）mostly safe (can conflict)
push本機 → remotelocal → remote不動本機；遠端會更新n/a local; remote updates安全；若 remote 領先會失敗safe; fails if remote is ahead
🛡️
新手規則：不確定時就先 git fetch——它永遠不會弄壞你的工作目錄。看清楚 git log HEAD..origin/main 多了什麼，再決定要 merge 還是 rebase。 Newcomer rule: when in doubt, git fetch first — it never modifies your working tree. Inspect what arrived with git log HEAD..origin/main, then decide whether to merge or rebase.
pull 策略

五、pull.rebase 該怎麼選？

Git 2.27 之後，第一次 git pull 會提醒你選擇策略：

Since Git 2.27, the first git pull asks which strategy to use:

# 三種策略 / Three options
$ git config --global pull.rebase false   # 預設：fetch + merge（產生 merge commit）
$ git config --global pull.rebase true    # fetch + rebase（線性歷史）
$ git config --global pull.rebase only    # 只允許 fast-forward，否則拒絕

false（merge）

新手友善；歷史保留分歧的「真實樣貌」。代價：每次 pull 都可能產生 merge commit，long-term repo 會出現很多「Merge branch 'main' of...」噪音。

Friendly for beginners; history preserves the "real shape" of divergence. Cost: every pull may produce a merge commit, and the log fills with "Merge branch 'main' of..." noise.

true（rebase）

線性歷史，git log 漂亮。代價：rebase 會改寫本機 commit hash，已 push 出去的東西別用。

Linear history, pretty git log. Cost: rebase rewrites local commit hashes — never use it on commits you've already pushed.

only

嚴格模式：只允許 fast-forward。一旦本機有未 push 的 commit 又 fetch 到新東西，就會拒絕——強迫你自己決定 merge 還是 rebase。

Strict: only fast-forward is allowed. If you have unpushed commits AND remote moved, it refuses — forcing you to decide merge vs rebase explicitly.

🎯
建議：新手選 false，熟練後改 only。true 適合單人開發或約定好「永遠 rebase」的小團隊。 Recommendation: beginners pick false; switch to only once you're comfortable. true suits solo work or a small team that has agreed "always rebase."

七、pull / push 出問題了怎麼辦？

🌳 同步失敗排查

Q1:
push 被拒絕，訊息為 rejected — fetch first？→ 是 → 先 git pull（或 fetch + merge），解完衝突再 git push。
Q2:
訊息為 non-fast-forward？→ 是 → 你的 push 和 remote 都動過。git pull --rebase 或 merge，再 push。
Q3:
authentication 失敗？→ 是 → ssh-add -l 看 key 在不在 agent；或檢查 PAT 是否過期、scope 是否足夠。
Q4:
只想安全地看 remote 有什麼新東西、不想 merge？→ 是 → git fetch，然後 git log HEAD..origin/main 看新 commit。
Q1:
Push was rejected, message says rejected — fetch first? → Yes → git pull (or fetch + merge), resolve conflicts, then git push.
Q2:
Message says non-fast-forward? → Yes → Both your branch and the remote moved. git pull --rebase or merge, then push.
Q3:
Authentication failed? → Yes → Check ssh-add -l for keys in the agent; or check whether your PAT expired or lacks scope.
Q4:
Just want to safely see what's new without merging? → Yes → git fetch, then git log HEAD..origin/main to inspect the new commits.
完整程式碼

常見場景指令套餐

# 1. 先在 GitHub 上建好空 repo（不要勾 README / .gitignore）
#    Create empty repo on GitHub first (no README/.gitignore)

# 2. 在本機 repo 內新增 remote / Add remote locally
$ git remote add origin git@github.com:charlene/scRNA-pipeline.git
$ git remote -v
origin  git@github.com:charlene/scRNA-pipeline.git (fetch)
origin  git@github.com:charlene/scRNA-pipeline.git (push)

# 3. 首次推上 main 並設定 upstream / First push, sets upstream
$ git push -u origin main

# 4. 之後就可以直接 / Later, simply:
$ git push
# 早上開工流程 / Morning workflow
$ git fetch origin                 # 看看遠端有沒有新東西，安全
$ git status                       # Your branch is behind origin/main by 3 commits
$ git log HEAD..origin/main --oneline   # 看是哪些新 commit

# 確認沒問題後拉下來 / Pull when satisfied
$ git pull                         # = fetch + merge (or rebase, 依設定)

# 寫完一段、commit、推 / Commit and push your work
$ git add analysis/qc.R
$ git commit -m "Add QC threshold tuning"
$ git push
# push 失敗訊息 / Push fails with:
#   ! [rejected]   main -> main (non-fast-forward)
#   error: failed to push some refs ...
#   hint: Updates were rejected because the tip of your current
#         branch is behind its remote counterpart.

# 1. 拉下來並 rebase 自己的 commit 到 remote 之上
$ git pull --rebase origin main

# 2. 若有衝突，解決後 / If conflicts, resolve then:
$ git add <resolved-files>
$ git rebase --continue

# 3. 再推一次 / Push again
$ git push

# 看 force push 之前先確認沒有別人的 commit 會被覆蓋
# Before any --force, double-check nobody else's commit gets erased
$ git push --force-with-lease    # 比 --force 安全
⚠️
不要對共用分支用 git push --force！它會把 remote 強制覆寫成你的版本，別人新推的 commit 會消失。要 force 也請用 --force-with-lease，並先跟團隊同步。 Never git push --force to a shared branch! It overwrites the remote with your version, erasing anyone else's recent pushes. If you must force-push, use --force-with-lease and coordinate with the team first.
🧬
生資情境：同時在本機 NB 和 HPC 開發時，git push 從 NB 上去，git pull 在 HPC 拉下來——把 Git 當成程式碼的 rsync。記得：data 用 rsync 或 LFS，code 用 git。 Bioinformatics: when juggling a laptop and HPC, git push from the laptop and git pull on HPC — Git becomes a poor man's rsync for code. Just remember: data still goes through rsync or Git LFS; code goes through Git.
