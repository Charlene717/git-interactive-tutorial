生資可重現研究 (Reproducibility)

Git 只解決了「程式碼在哪？」這個問題——這一章告訴你剩下 80% 怎麼補齊：環境、容器、資料、文件、典藏。

Git solves "where's the code?" — this final chapter covers the other 80%: environments, containers, data, documentation, and archival.

概覽

可重現研究的四個層次

版本控制只是可重現研究的一根支柱。Peng (2011) 與 Sandve (2013) 把可重現性拆成四個層次，由易到難：

  • 同程式、同資料、同結果（可重跑）——最低門檻，但很多論文連這都做不到。
  • 同程式、不同資料、預期結果（穩健性）——能換一批樣本仍給出合理輸出。
  • 同程式、未來電腦、同結果（運算可重現）——五年後的人能不能跑出來？
  • 另一組人獨立驗證（可複製）——黃金標準。

Git/GitHub 解決「程式碼在哪」；本章處理剩下 80%：環境、容器、資料、文件與長期典藏。

Version control is just one pillar of reproducible research. Peng (2011) and Sandve (2013) define four levels of reproducibility, from easiest to hardest:

  • Same code, same data, same result (re-runnable) — the floor, but many papers fail even here.
  • Same code, different data, expected result (robust) — does it still behave sensibly on a different sample set?
  • Same code, future computer, same result (computationally reproducible) — can someone rerun it in five years?
  • Verified independently by another team (replicable) — the gold standard.

Git/GitHub answers "where's the code?"; this chapter covers the other 80%: environments, containers, data, documentation, and archival.

💡
實用原則：你不需要一次達到第四層。先確保「同人、同機、半年後」還能重跑——這已經淘汰掉超過一半的論文了。 Practical principle: you don't have to hit level four on day one. Just ensure that you yourself can rerun the analysis on the same machine six months later — that alone already beats half the published literature.

環境鎖定

一、鎖定環境（conda / pip / renv）

工具 鎖定指令 特性
conda (lite)conda env export --from-history > environment.yml只記錄你明確安裝的套件，跨 OS 可用，推薦。Records only what you explicitly installed; cross-OS portable; recommended.
conda (full)conda env export > environment.yml過度鎖定（含 OS-specific build hash），同事換 OS 就掛。Over-pins with OS-specific build hashes; breaks if a collaborator switches OS.
conda-lockconda-lock -f environment.yml產生每個 OS 的精確 lockfile，重現性最高。Generates per-OS exact lockfiles — highest fidelity reproduction.
pippip freeze > requirements.txt或用 pip-compile / Poetry / uv 鎖定。Or use pip-compile / Poetry / uv for proper locking.
R / renvrenv::snapshot()寫出 renv.lock，含精確版本與 CRAN/Bioconductor 來源。renv::restore() 重建。Writes renv.lock with exact versions and CRAN/Bioconductor sources. renv::restore() rebuilds.
⚠️
務必 commit lockfile：沒進 Git，等於沒做。 environment.yml、renv.lock、requirements.txt 都要進 repo。 Always commit lockfiles: if it's not in Git, it doesn't exist. Commit environment.yml, renv.lock, and requirements.txt to the repo.
容器

二、容器：Docker & Singularity

🐳 Docker

把 OS、工具、程式碼打包成不可變的映像檔（image），標識為 SHA digest。同樣的映像在任何 Docker 環境下都跑出相同結果。適合本地、雲端、CI。

Bundles OS + tools + code into an immutable image identified by a SHA digest. The same image yields the same result in any Docker environment. Great for laptops, cloud, and CI.

🛡️ Singularity / Apptainer

HPC 友善：不需要 root 也能跑（Docker 要 root，學校 cluster 通常不給）。可直接從 Docker Hub 拉映像並轉換，學術圈最常見。

HPC-friendly: runs without root (Docker requires root, which academic clusters rarely grant). Can pull and convert Docker images directly; the de facto standard in academia.

標準工作流程：(1) 在 repo 內寫 Dockerfile；(2) docker build 建立映像；(3) push 到 Docker Hub 或 GitHub Container Registry (GHCR)；(4) pipeline config 指定映像版本。配合 Git tag——Dockerfile 在 tag v1.0.0，映像也 tag 為 myrepo:1.0.0，任何人都能重建。

Standard workflow: (1) write a Dockerfile in the repo; (2) docker build to create the image; (3) push to Docker Hub or GitHub Container Registry (GHCR); (4) reference the image version in the pipeline config. Pair with Git tags — Dockerfile lives at tag v1.0.0, image is tagged myrepo:1.0.0, anyone can recreate.

資料與雜湊

三、資料與檔案完整性 (checksums)

Step 9 已經說過：原始定序資料絕對不要進 Git。但你必須清楚標示資料在哪、版本是什麼：

  • 公開資料：引用 accession number——SRA (SRP123456)、GEO (GSE171524)、ENA、ArrayExpress、TCGA、HCA。
  • 自製資料：投稿前同步存到 Zenodo / Figshare 並取得 DOI，論文引用 DOI。
  • 檔案 checksums：把 md5sum 或 sha256sum 結果存在 data/checksums.txt，任何下載者都能驗證檔案是否完整。

From Step 9: raw sequencing data must never enter Git. But you must declare where the data lives and what version it is:

  • Public data: cite the accession — SRA (SRP123456), GEO (GSE171524), ENA, ArrayExpress, TCGA, HCA.
  • Your own data: before submission, deposit on Zenodo / Figshare to obtain a DOI; the paper cites the DOI.
  • File checksums: store md5sum or sha256sum output in data/checksums.txt; anyone downloading can verify integrity.
# 產生 checksums / Generate checksums
$ sha256sum data/raw/*.fastq.gz > data/checksums.txt

# 下載者驗證 / Verify after download
$ sha256sum -c data/checksums.txt
data/raw/sample01_R1.fastq.gz: OK
data/raw/sample01_R2.fastq.gz: OK
引用與典藏

四、CITATION.cff 與 Zenodo DOI

把 CITATION.cff（Citation File Format，YAML）放在 repo 根目錄，GitHub 會自動在 repo 頁面側邊欄產生「Cite this repository」按鈕，輸出 BibTeX / APA 等格式。

Place a CITATION.cff (Citation File Format, YAML) at the repo root and GitHub auto-renders a "Cite this repository" button on the side panel that produces BibTeX / APA / etc.

# CITATION.cff 範例 / Example
cff-version: 1.2.0
message: "If you use this software, please cite as below."
authors:
  - family-names: Yang
    given-names: Charlene
    orcid: https://orcid.org/0000-0000-0000-0000
title: "scRNA-seq Pipeline"
version: 1.4.2
doi: 10.5281/zenodo.12345678
date-released: 2026-05-14

Zenodo + GitHub 整合：到 zenodo.org 啟用整合，勾選想要典藏的 repo——之後每個 GitHub Release 都會自動送一份到 Zenodo 並取得 DOI。論文引用這個 DOI（不要引 GitHub URL，URL 可能改）。

Zenodo + GitHub integration: visit zenodo.org, enable the integration, and toggle the repos you want archived. Every subsequent GitHub Release is automatically snapshotted to Zenodo with its own DOI. Your paper cites the DOI — never a bare GitHub URL, which can move.

💡
時機：論文被接受後，下 git tag v1.0.0、gh release create v1.0.0——Zenodo 自動發 DOI。論文 Methods 寫「Code: 10.5281/zenodo.XXXXX」就完成了。 Timing: when the paper is accepted, git tag v1.0.0, then gh release create v1.0.0 — Zenodo issues the DOI automatically. Your Methods section just says "Code: 10.5281/zenodo.XXXXX" and you're done.

README 範本

五、讓別人能重現的 README

好的 README 是「論文外的補充材料」，要回答這些問題：

  1. 這是什麼？一段話描述用途與輸入/輸出。
  2. 怎麼安裝？一行：conda env create -f environment.yml 或 docker pull myrepo:1.0.0。
  3. 怎麼快速試？一條指令跑通一個小範例。
  4. 資料從哪來？accession + DOI。
  5. 怎麼引用？BibTeX 或 Zenodo DOI。
  6. 授權？學術程式碼用 MIT 或 BSD-2（寬鬆）；要強制下游 open source 用 GPL-3；想加上專利條款用 Apache-2。

A good README is the "supplementary material" to your paper. It must answer:

  1. What is this? A short paragraph on purpose and I/O.
  2. How do I install? One-liner: conda env create -f environment.yml or docker pull myrepo:1.0.0.
  3. Quick start? One command runs a small test case.
  4. Where's the data? Accessions + DOI.
  5. How do I cite? BibTeX or Zenodo DOI.
  6. License? Academic code: MIT or BSD-2 (permissive). Want copyleft? GPL-3. Want a patent clause? Apache-2.

完整程式碼範例

# --- conda (portable) ---
$ conda env export --from-history > environment.yml
$ git add environment.yml && git commit -m "chore: pin conda env"

# --- conda-lock (exact, per-OS) ---
$ conda-lock -f environment.yml -p linux-64 -p osx-arm64

# --- R / renv ---
> renv::init()
> renv::snapshot()    # writes renv.lock
> renv::restore()     # colleague rebuilds env

# --- Dockerfile skeleton ---
FROM mambaorg/micromamba:1.5
COPY environment.yml /tmp/env.yml
RUN micromamba install -y -n base -f /tmp/env.yml && \
    micromamba clean -a -y
COPY . /pipeline
WORKDIR /pipeline
CMD ["snakemake", "--cores", "4"]
# 1. 標記 release / Tag a release
$ git tag -a v1.0.0 -m "v1.0.0 — paper submission snapshot"
$ git push origin v1.0.0

# 2. 用 gh CLI 建立 GitHub Release（觸發 Zenodo）
$ gh release create v1.0.0 \
    --title "v1.0.0 — paper submission" \
    --notes-file CHANGELOG.md

# 3. Zenodo 自動歸檔 → DOI
#   檢查 https://zenodo.org/account/settings/github/
#   找到 repo 對應的 DOI 徽章，貼回 README

# 4. 更新 CITATION.cff 加上 doi 與 version
$ git commit -am "docs: update CITATION.cff with v1.0.0 DOI"
# scRNA-seq Pipeline

A Snakemake pipeline for single-cell RNA-seq QC, normalization, and clustering.

## Install

```bash
conda env create -f environment.yml
conda activate scrna-pipe
```

Or with Docker:

```bash
docker pull ghcr.io/charlene/scrna-pipe:1.0.0
```

## Quick start

```bash
snakemake --use-conda --cores 8 --config sample=test
```

## Data

- Test data: GEO accession GSE171524
- Custom dataset: Zenodo DOI 10.5281/zenodo.12345678

## Citation

If you use this pipeline, please cite:
> Yang C. (2026). scRNA-seq Pipeline (v1.0.0). Zenodo. https://doi.org/10.5281/zenodo.12345678

See CITATION.cff for BibTeX.
