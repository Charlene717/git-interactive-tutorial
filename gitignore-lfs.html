.gitignore & Git LFS（生物資訊版）

FASTQ、BAM、H5AD 都不該丟進 Git——學會 .gitignore 規則、Git LFS 的取捨，以及該存到哪。

FASTQ, BAM, H5AD — none of these belong in Git. Learn the .gitignore rules, Git LFS trade-offs, and where the data should actually live.

概覽

為什麼基因體資料不能直接 commit？

Git 當初是為了管理 Linux 核心的原始碼而設計——幾百萬行純文字。它不是為了存 50 GB 的 BAM 檔而生的。沒有規劃就 git add . 的下場：(1) 不小心把一個 50 GB BAM 加進去，(2) repo 體積爆炸到無法 clone，(3) GitHub 直接拒絕你的 push（單一檔案上限 100 MB、push 上限 2 GB）。

本章涵蓋四件事：(1) 什麼絕對不要 commit；(2) .gitignore 規則語法；(3) Git LFS 何時值得開、何時不值得；(4) 最重要的替代方案——不要追蹤資料，只追蹤產生資料的程式碼與 accession 編號。

Git was originally designed to manage the Linux kernel's source code — millions of lines of plain text. It was not built to hold 50 GB BAM files. Run git add . without thinking and you get: (1) a 50 GB BAM accidentally staged, (2) a repo so bloated nobody can clone it, (3) GitHub rejecting your push (100 MB per-file limit, 2 GB per-push limit).

This chapter covers four things: (1) what you must never commit; (2) .gitignore pattern syntax; (3) when Git LFS makes sense and when it doesn't; (4) the most important alternative — don't track the data at all, only track the code that generates it and the accession numbers that point to it.

💡
核心心法：Git 追蹤程式碼，資料用 SRA / Zenodo / S3 / NAS 存。論文 reviewer 看到的 repo，應該是「clone 下來、跑 pipeline、複現我的結果」——不是「請先下載 80 GB 才能開始」。 Mental model: Git tracks code; data lives in SRA, Zenodo, S3, or your NAS. The repo a reviewer clones should be "clone → run pipeline → reproduce my results" — not "first, download 80 GB of data."

語法

.gitignore 規則語法

📐 基本 pattern

# 萬用字元 / wildcard
*.bam              # 所有 .bam
data/              # 整個資料夾
data/raw/*.fq.gz   # 子目錄特定檔
**/temp/           # 任何深度的 temp/
!keep.csv          # 否定：例外保留
.DS_Store          # 隱藏檔

📍 放在哪？

通常放 repo 根目錄；子目錄也可以放各自的 .gitignore（規則只影響該子樹）。.gitignore 本身要被 commit——這樣協作者 clone 後也繼承同一份 ignore 規則。

Usually at the repo root; subdirectories can also have their own .gitignore (scoped to that subtree). The .gitignore file itself must be committed so collaborators inherit the same rules on clone.

🌐
GitHub 在 github.com/github/gitignore 維護各種語言的 .gitignore 模板。新 repo 時 GitHub 介面也會問你要不要套用——R、Python 都有現成版本，直接套再加生資專用規則最快。 GitHub maintains language-specific .gitignore templates at github.com/github/gitignore. When creating a new repo, GitHub also offers them in the UI — start with R or Python, then layer on your bioinformatics-specific rules.

⚠️
順序很重要：否定規則（!）必須在它要例外的 pattern 之後。例如 results/*.pdf + !results/figures/final_figure_1.pdf，順序顛倒會失效。 Order matters: a negation rule (!) must come after the pattern it excepts. E.g. results/*.pdf followed by !results/figures/final_figure_1.pdf — reverse the order and the negation has no effect.
不小心 commit 了大檔，怎麼救？

已經 commit / push 才發現怎麼辦？

情境 A：剛 commit、還沒 push。從 index 移除但保留磁碟上的檔案，加進 .gitignore，再 commit。

Scenario A: just committed, not pushed yet. Remove from the index but keep the file on disk, add it to .gitignore, then commit.

$ git rm --cached path/to/big.bam
$ echo "*.bam" >> .gitignore
$ git add .gitignore
$ git commit -m "Stop tracking BAM files"

情境 B：已經 push、需要從歷史中徹底清除。使用 git filter-repo（現代推薦）或 BFG Repo-Cleaner。注意：這會改寫歷史，所有協作者必須重新 clone。

Scenario B: already pushed, need to purge from history. Use git filter-repo (modern, recommended) or the BFG Repo-Cleaner. This rewrites history, so every collaborator must re-clone.

# 移除歷史中所有 large.bam / Remove large.bam from all history
$ git filter-repo --path large.bam --invert-paths

# 強制 push 改寫後的歷史 / Force-push the rewritten history
$ git push origin --force --all
$ git push origin --force --tags
🚨
GitHub 拒絕 > 100 MB 的單檔！ 一旦不小心 push，你必須先 rewrite history 才能再次 push。所以最好的策略是事前 .gitignore 寫好，而不是事後挽救。 GitHub rejects any push containing a file > 100 MB. Once you push it by accident, you must rewrite history before another push will succeed. The best strategy is to write .gitignore before staging — not to rescue afterwards.
Git LFS

Git LFS 是什麼？什麼時候用？

Git LFS (Large File Storage)：把大檔存到外部 server，repo 內只留一個小小的 pointer file（一行 hash）。Clone 時 LFS client 才會去下載真正的檔案。

Git LFS (Large File Storage) stores big files outside Git on an LFS server; your commits only contain a tiny pointer file (a single hash). The LFS client downloads the real file on clone or checkout.

# 安裝（每台機器一次） / Install (once per machine)
$ git lfs install

# 告訴 LFS 追蹤特定副檔名 / Track a file pattern
$ git lfs track "*.h5"
$ git lfs track "models/*.pkl"

# 上面指令會更新 .gitattributes — 別忘了 commit 它
$ git add .gitattributes
$ git commit -m "Track *.h5 via LFS"

✅ LFS 適合的情況

  • 檔案必須跟程式碼一起版本化（例如參考索引、訓練好的 model checkpoint、論文 supplementary）
  • 單檔 < 5 GB
  • 總量符合 GitHub LFS 配額（免費 1 GB）
  • The file must be versioned alongside code (e.g. reference index, trained model checkpoint, paper supplementary data)
  • Single file < 5 GB
  • Total fits GitHub LFS quota (1 GB free)

❌ LFS 不適合的情況

  • 原始 FASTQ — 太大、應該在 SRA
  • 公開資料集 — 用 accession 連結即可
  • 可以由程式重新產生的檔案（圖、表）
  • Raw FASTQs — too big, belong in SRA
  • Public datasets — just link the accession
  • Anything regeneratable from code (figures, tables)
💸
GitHub LFS 配額地雷：免費帳號只有 1 GB 儲存 + 1 GB / 月頻寬——而且是「整個帳號」共享，不是每個 repo。超過後必須買 data pack（每包 $5/月，5 GB 儲存 + 50 GB 頻寬）。把 model 放進 LFS 前先估算流量。 GitHub LFS quota gotcha: a free account gets only 1 GB of storage + 1 GB/month of bandwidth — and the limits are per account, not per repo. Beyond that you must buy data packs ($5/month each, 5 GB storage + 50 GB bandwidth). Estimate traffic before putting a model in LFS.

🌳 逐項判斷

Q1:
是原始碼（script、config、文件）？→ 是 → 正常 commit。
Q2:
是原始或中間資料（FASTQ、BAM、h5ad）？→ 是 → 絕對不 commit；放外部儲存、README 寫 accession。
Q3:
是小型衍生結果（CSV 表、≤ 5 MB 圖）？→ 是 → 可以 commit，但優先考慮從程式重新產生。
Q4:
是必須版本化的二進位檔且 < 1 GB？→ 是 → 用 Git LFS。
Q5:
是 Jupyter notebook（.ipynb）？→ 先做 → nbstripout 清掉 output 再 commit。
Q1:
Is it source code (script, config, docs)? → Yes → Commit normally.
Q2:
Is it raw or intermediate data (FASTQ, BAM, h5ad)? → Yes → Never commit. Use external storage; cite accessions in README.
Q3:
Small derived result (CSV table, figure ≤ 5 MB)? → Yes → You can commit it — but prefer regenerating from code.

# 加上生資專用規則 / Append bioinformatics-specific rules

*.fastq.gz
*.bam
*.h5ad
.snakemake/

EOF

$ git add .gitignore
$ git commit -m "Add bioinformatics .gitignore"
# 1. 安裝 LFS（每台機器一次） / Install LFS once per machine
$ git lfs install

# 2. 告訴 LFS 要追蹤哪些 pattern / Track patterns
$ git lfs track "*.h5"
$ git lfs track "models/*.pt"

# 3. commit .gitattributes（這是 LFS 自動產的）
$ git add .gitattributes
$ git commit -m "Track *.h5 and model checkpoints via LFS"

# 4. 加入大檔，它會自動存進 LFS
$ git add models/best.pt
$ git commit -m "Add trained model checkpoint"
$ git push origin main

# 確認 LFS 真的接管了哪些檔 / Verify which files LFS handles
$ git lfs ls-files
# A. 還沒 push：從 index 移除、留磁碟、加進 .gitignore
$ git rm --cached huge.bam
$ echo "*.bam" >> .gitignore
$ git add .gitignore
$ git commit -m "Stop tracking BAM files"

# B. 已 push：從歷史完整清除（會改寫 history）
# 安裝 git-filter-repo: pip install git-filter-repo
$ git filter-repo --path huge.bam --invert-paths

# 強制 push、告知所有協作者重新 clone
$ git push origin --force --all
$ git push origin --force --tags

# 釋放本地 .git 物件空間
$ git reflog expire --expire=now --all
$ git gc --prune=now --aggressive
⛔
filter-repo 是不可逆操作！它改寫每個 commit 的 SHA，所有人之前 clone 的本地 repo 都失效，必須重新 clone。動手前先在 Slack / email 告知所有協作者，並備份 repo（git clone --mirror）。 filter-repo is irreversible. It rewrites every commit SHA, invalidating every collaborator's local clone — they must re-clone. Coordinate on Slack/email before running it, and back up the repo first with git clone --mirror.
🧬
生資現場慣例：FASTQ 原始資料屬於 SRA / ENA / GEO。論文 repo 的 README 只要列出 accession（如 GSE123456、SRP456789）和 fasterq-dump 指令，reviewer 自己拉就好——不要把 80 GB 塞進 GitHub。 Bioinformatics convention: raw FASTQs belong in SRA / ENA / GEO. Your paper repo's README should just list the accessions (e.g. GSE123456, SRP456789) and a fasterq-dump command — let the reviewer fetch the data themselves. Don't push 80 GB to GitHub.
