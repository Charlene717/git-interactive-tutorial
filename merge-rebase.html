STEP 6 / 9

合併與 Rebase (Merging & Rebasing)

分支開好之後，要把工作整合回來——merge 保留歷史，rebase 重寫歷史，cherry-pick 是外科手術；衝突總會出現，學會冷靜處理。

Once you've branched, you have to integrate work back. Merge preserves history; rebase rewrites it; cherry-pick is surgical. Conflicts will happen — learn to handle them calmly.

概覽

為什麼整合分支是門學問？

分支只是準備工作；真正的價值是把分支裡的成果整合回主線。Git 提供兩條主路：merge（保留兩條分支的完整歷史，並用一個 merge commit 把它們接起來）與 rebase（把你的 commit「搬」到目標分支頂端，看起來像線性開發）。各有取捨。

還有外科手術用的 cherry-pick：只把某幾個 commit 複製到另一條分支（例如把 hotfix 同時套到 main 和 release/v1）。最後，無論用哪一種，衝突遲早會發生——當兩條分支改了同一行，Git 沒辦法自動決定誰對，這時候要你手動處理。

Branching is just preparation; the value comes from integrating work back into the mainline. Git gives you two main paths: merge (preserves both branches' history and stitches them with a merge commit) and rebase (replays your commits on top of the target branch so history looks linear). Each has trade-offs.

For surgical operations there's cherry-pick: copy specific commits from one branch onto another (handy when a hotfix needs to land on both main and release/v1). And whichever path you choose, conflicts will eventually happen — when both branches edit the same line, Git can't decide which version wins, and asks you to resolve it.

💡
核心心法：merge 永遠是「加法」（不破壞歷史）；rebase 是「重寫」（產生新的 SHA）。學會看 git log --graph --oneline --all，你就能從圖形分辨兩種結果。 Core idea: merge is always additive (never rewrites history); rebase replays and produces brand-new SHAs. Learn to read git log --graph --oneline --all — its shape tells you instantly which strategy was used.
MERGE 基礎

一、merge 的三種樣貌

➡️

Fast-forward (ff)

主分支自從你開分支後沒有再前進。Git 只需把指標往前挪——不會產生新的 commit，歷史完全線性。

The target branch hasn't moved since you branched off. Git simply slides the pointer forward — no merge commit, perfectly linear history.

🔀

3-way merge

兩條分支都各自前進過。Git 找到共同祖先 + 兩端，自動產生一個有「兩個父 commit」的 merge commit。

Both branches advanced. Git finds the common ancestor plus both tips and creates a merge commit with two parents.

📦

no-ff merge

明明可以 ff，但你用 --no-ff 強迫產生 merge commit——讓「這幾個 commit 屬於同一個 feature」在歷史上看得出來。

Even when ff is possible, --no-ff forces a merge commit so the history visibly groups "these commits belonged to one feature."

# ff merge（target 沒前進過） / Fast-forward
$ git switch main
$ git merge feature
Updating a1b2c3..d4e5f6
Fast-forward

# 強制保留分支痕跡 / Always create a merge commit
$ git merge --no-ff feature -m "Merge feature/qc-filter into main"

# 把 feature 的所有 commit 壓成一個 staged 變更（尚未 commit）
$ git merge --squash feature
$ git commit -m "Add QC filter module (squashed 8 WIP commits)"

ASCII 對照

--- 之前 / Before ---
main:    A---B---C
                  \
feature:           D---E

--- ff merge ---           --- 3-way merge (no-ff) ---
main:    A---B---C---D---E      main:    A---B---C-------M   (M 有兩個父)
                                                  \     /
                                feature:           D---E

--- squash merge ---
main:    A---B---C---S          (S 包含 D+E 的內容，但只有一個父，feature 分支不在歷史上)
REBASE 基礎

二、rebase：把 commit 搬家

git rebase 的本質是「把你這邊的 commit 一個一個重新套用到新的基底上」。原本的 commit 並沒有被改——而是被放棄了，Git 用相同的變更內容 + 新的父節點，產生新的 SHA。從 git log 看會以為時間順序變了，事實上是整段歷史被重寫了。

git rebase takes each of your commits and re-applies it on top of a new base. The originals aren't modified — they're abandoned, and Git creates new commits (same diffs, new parent) with brand-new SHAs. Reading git log afterward, it looks like time was reordered; in reality, that segment of history has been rewritten.

# 1. 標準 rebase（在 feature 上，把 commits 搬到最新 main 之上）
$ git switch feature
$ git rebase main
# Successfully rebased and updated refs/heads/feature.

# 2. 互動式 rebase：清理最近 5 個 commit
$ git rebase -i HEAD~5
# 編輯器會打開，每行一個 commit，動作關鍵字可改：
#   pick   = 保留
#   reword = 改 commit message
#   edit   = 暫停讓你改檔案
#   squash = 合併到上一個 commit (保留 messages)
#   fixup  = 合併到上一個 commit (丟棄 message)
#   drop   = 刪除這個 commit
#   調換順序 = 直接剪下貼上整行
⛔
黃金法則（Golden Rule of Rebase）：絕對不要對已經 push 到共用分支的 commit 做 rebase。rebase 會改 SHA，協作者本地的 history 還是舊 SHA，下次 pull 會打架；運氣不好整個 repo 都會錯亂。Rebase 只用在自己的、還沒 push 的 branch。 The golden rule of rebase: never rebase commits you've already pushed to a shared branch. Rebase changes the SHAs, your collaborators still hold the old ones, and their next pull will fight you. Rebase is for your own branches that haven't been pushed.

CHERRY-PICK

四、cherry-pick：只搬一個 commit

git cherry-pick <sha> 把另一條分支上的某一個（或幾個）commit「複製」到目前分支。最常見的場景：hotfix 在 main 寫完，要同時套到 release/v1 分支上——cherry-pick 是最乾淨的做法。

git cherry-pick <sha> copies one (or several) commits from another branch onto your current branch. Classic use case: a hotfix landed on main needs to be applied to release/v1 too — cherry-pick is the cleanest tool.

# 把單一 commit 套到目前分支 / Apply a single commit
$ git switch release/v1
$ git cherry-pick a1b2c3d

# 一次套多個 / Apply multiple commits
$ git cherry-pick a1b2c3d e4f5g6h

# 套一個區間（不含 start，包含 end） / A range
$ git cherry-pick a1b2c3d..e4f5g6h

# -x: 在 commit message 裡記下原始 SHA（建議！）
$ git cherry-pick -x a1b2c3d
# 結果 message 多一行：(cherry picked from commit a1b2c3d)

# -n: 只套變更，不自動 commit（讓你修改後再 commit）
$ git cherry-pick -n a1b2c3d
衝突解決

五、衝突來了怎麼辦？

當兩條分支改了同一個檔案的同一行，merge / rebase / cherry-pick 都會暫停。git status 會顯示 both modified: path/to/file，檔案內部會出現衝突標記：

When two branches edit the same line of the same file, merge/rebase/cherry-pick all pause. git status shows both modified: path/to/file and the file itself contains conflict markers:

# 衝突標記長這樣 / Conflict markers
<<<<<<< HEAD
threshold = 0.05      # 目前分支（main）的版本
=======
threshold = 0.01      # 另一邊（feature）的版本
>>>>>>> feature/strict-qc

# 標準解決流程 / Standard workflow
# 1. 打開檔案，手動編輯（選擇/合併兩邊內容、刪掉 <<< === >>> 標記）
# 2. 標記為已解決
$ git add path/to/file
# 3a. 如果是 merge：
$ git commit       # Git 已準備好預設的 merge message
# 3b. 如果是 rebase：
$ git rebase --continue
# 3c. 如果是 cherry-pick：
$ git cherry-pick --continue

# 放棄整個操作 / Abort and bail out
$ git merge --abort
$ git rebase --abort
$ git cherry-pick --abort
🛠️
工具加持：git mergetool 可呼叫 vimdiff / meld / kdiff3 等三面板比較工具。VS Code 內建衝突解決介面，按鈕直接選「Accept Current」「Accept Incoming」「Accept Both」，比手動刪標記快很多。 Tooling helps: git mergetool launches a three-pane diff (vimdiff / meld / kdiff3). VS Code's built-in conflict UI gives you "Accept Current / Incoming / Both" buttons — much faster than deleting markers by hand.

決策

七、該 merge 還是 rebase？

🌳 merge 或 rebase 決策樹

Q1:
這些 commit 已經 push 到共用分支了嗎？→ 是 → 一律 merge（rebase 會破壞別人的歷史）。
Q2:
要保留「這是分支的工作」這個事實？→ 是 → 用 merge --no-ff，讓分支痕跡留在歷史上。
Q3:
你的 local 有一堆 WIP 雜亂 commit，想在開 PR 前整理乾淨？→ 是 → rebase -i 把它們壓成一兩個有意義的 commit。
Q4:
團隊政策要求 main 上不能有 merge commit？→ 是 → 用 rebase 或在 GitHub 上選 Squash and merge。
Q1:
Have those commits been pushed to a shared branch? → Yes → Always merge (rebase would break others' history).
Q2:
Do you want history to show "this work belonged to a branch"? → Yes → Use merge --no-ff to keep the branch shape visible.
Q3:
You have messy WIP commits locally and want to tidy them up before opening a PR? → Yes → Use rebase -i to squash them into one or two meaningful commits.
Q4:
Team policy forbids merge commits on main? → Yes → Use rebase, or pick "Squash and merge" on GitHub.
⚠️
不確定時——用 merge。merge 永遠是安全的；rebase 是 power tool，請只在自己的 private branch 上使用。 When in doubt, merge. Merge is always safe; rebase is a power tool — reserve it for your own private branches.

八、常用情境程式碼

# 把 feature 合回 main / Merge feature back into main
$ git switch main
$ git pull                              # 確認 main 是最新
$ git merge --no-ff feature             # 保留分支結構
# 編輯 merge commit message → 儲存 → 完成

# 推上去 / Push
$ git push origin main

# 清理 / Clean up the local branch
$ git branch -d feature                 # -d 只刪已 merge 的；安全
# 若還要刪遠端分支
$ git push origin --delete feature
# 在 PR 開出去之前，把 feature rebase 到最新 origin/main 之上
$ git switch feature
$ git fetch origin
$ git rebase origin/main

# 若途中有衝突——解完 → add → continue
# 全部結束後（注意：歷史被改了，必須 force-push）
$ git push --force-with-lease origin feature
#         ^^^^^^^^^^^^^^^^^ 比 --force 安全：
#         如果遠端有別人新的 commit 你不知道，會拒絕

# ⚠️ 只在「只有你自己」的分支上 force-push！
# 假設正在 merge，發生衝突
$ git merge feature
Auto-merging analysis/qc.py
CONFLICT (content): Merge conflict in analysis/qc.py
Automatic merge failed; fix conflicts and then commit.

# 1. 看看哪些檔案有衝突
$ git status
        both modified:   analysis/qc.py

# 2. 編輯檔案：把 <<<HEAD ... === ... >>>feature 改成最終想要的內容
$ code analysis/qc.py

# 3. 標記為已解決
$ git add analysis/qc.py

# 4. 完成 merge（rebase 用 --continue，cherry-pick 同樣）
$ git commit                 # merge
# 或
$ git rebase --continue
$ git cherry-pick --continue

# 不想處理了 → 全部退回原狀
$ git merge --abort
# 整理最近 5 個 commit / Clean up last 5 commits
$ git rebase -i HEAD~5

# 編輯器會彈出（從舊到新）:
pick   a1b2c3d  WIP: add scaffold
pick   d4e5f6g  WIP: fix typo
pick   h7i8j9k  WIP: more fixes
pick   l0m1n2o  WIP: actually working now
pick   p3q4r5s  WIP: cleanup

# 改成這樣（4 個 fixup 進第一個，並重新命名）：
reword a1b2c3d  feat(qc): add adaptive QC threshold module
fixup  d4e5f6g  WIP: fix typo
fixup  h7i8j9k  WIP: more fixes
fixup  l0m1n2o  WIP: actually working now
fixup  p3q4r5s  WIP: cleanup
# → 儲存 → Git 自動執行，最後變成一個漂亮 commit

# 其他常用動作:
#   edit   = 暫停讓你改檔案/拆 commit
#   squash = 合併且保留 message（與下方合）
#   drop   = 丟掉這個 commit
#   調換 pick 行的順序 = 重新排序 commit

📝 自我檢測

1. 你目前在 feature 分支上，執行 git rebase main。會發生什麼事？

1. You're on feature and run git rebase main. What happens?

A. main 的 commit 被搬到 feature 之上，main 上的 SHA 全變A. main's commits are moved on top of feature, and main's SHAs change
B. feature 的 commit 被重新套用到最新 main 的頂端，並產生新的 SHAB. Your feature commits are re-applied on top of the latest main, with new SHAs
C. 產生一個 merge commit，feature 和 main 都不變C. A merge commit is created; neither branch changes
D. main 被切換為目前分支D. main becomes the current branch

2. merge 時出現衝突，正確的處理順序是？

2. A conflict appears during merge. Correct sequence?

A. 立刻執行 git merge --abort，重 cloneA. Run git merge --abort immediately and re-clone
B. 直接 git commit -m "fix" 略過衝突B. Run git commit -m "fix" and skip the conflict
C. 編輯檔案、刪除衝突標記、git add，再 git commit（或 git rebase --continue）C. Edit the file, remove markers, git add, then git commit (or git rebase --continue)
D. 用 git reset --hard 把衝突檔案蓋掉D. Run git reset --hard to overwrite the conflicted file

3. 你昨天已經把 feature push 上 GitHub，今天用 rebase -i 把 5 個 commit squash 成 1 個，然後再 push。結果？

3. You pushed feature to GitHub yesterday; today you squash 5 commits into 1 with rebase -i and push again. Result?

A. 正常 push，無副作用A. Pushes normally, no side effects
B. Git 自動合併新舊歷史B. Git auto-merges old and new history
C. 需要 force-push；如果別人也基於這個分支工作，他們的 history 會錯亂——對共用分支是壞習慣C. You need a force-push; if anyone else based work on that branch, their history breaks — bad practice on shared branches
D. 上游分支會被刪除D. The upstream branch will be deleted
