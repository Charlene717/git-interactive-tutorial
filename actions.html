GitHub Actions & CI/CD（生物資訊版）

讓 GitHub 幫你跑測試、檢查 pipeline、發佈 Docker image——code 一推上去，自動驗證就開跑。

Let GitHub run your tests, validate pipelines, and ship Docker images — every push triggers automated checks.

概覽

什麼是 CI / CD？為什麼生資專案需要？

CI (Continuous Integration，持續整合)：每次 push 程式碼時自動跑測試，第一時間抓到問題。CD (Continuous Deployment/Delivery，持續部署/交付)：通過測試後自動發佈版本、打包 Docker image、或部署文件網站。

GitHub Actions 是 GitHub 內建的 CI 服務——公開 repo 完全免費，私有 repo 每月 2000 分鐘額度。對生資專案來說：每個 PR 都用一份小型測試資料跑一次 pipeline，能在 code 進實驗室伺服器前就抓到 regression。

CI (Continuous Integration): automatically run tests on every push so you catch breakage immediately. CD (Continuous Deployment/Delivery): automatically publish releases, build Docker images, or deploy docs after tests pass.

GitHub Actions is GitHub's built-in CI service — free for public repos, generous for private. For bioinformatics: run a small test dataset through your pipeline on every PR to catch regressions before they reach the lab server.

💡
核心比喻：CI 是「自動 code reviewer 永遠不下班」。寫好測試後，每次 push 都會被檢查——你不用記得跑、合作者也不用提醒你。 Core analogy: CI is the code reviewer that never clocks out. Once tests exist, every push is checked — you don't have to remember, and your collaborators don't have to nag you.
核心概念

六個一定要懂的詞彙

📄

Workflow

放在 .github/workflows/ 的 YAML 檔。定義何時觸發（push / PR / 排程）以及要做什麼（jobs）。

YAML file in .github/workflows/. Defines when (push / PR / schedule) and what (jobs).

⚙️

Job

在 runner（Ubuntu / macOS / Windows VM）上執行的一個工作單位。多個 job 可以平行跑。

A unit running on a runner (Ubuntu / macOS / Windows VM). Multiple jobs run in parallel.

🪜

Step

單一指令或 action：uses: actions/checkout@v4 或 run: pytest。

A single command or action: uses: actions/checkout@v4 or run: pytest.

🖥️

Runner

實際跑 workflow 的 VM。可用 GitHub 主機，或 self-hosted（例如架在你的實驗室 HPC 上）。

The VM that runs the workflow. GitHub-hosted, or self-hosted (e.g., on your lab's HPC).

🎯

Trigger

on: push、on: pull_request、on: schedule（cron）、on: workflow_dispatch（手動）。

on: push, on: pull_request, on: schedule (cron), on: workflow_dispatch (manual).

🔐

Secret

加密 env 變數（API key、deploy key）。存在 repo Settings → Secrets。用 ${{ secrets.NAME }} 引用。

Encrypted env var (API keys, deploy keys). Stored in repo Settings → Secrets. Reference via ${{ secrets.NAME }}.

最小範例

第一個 workflow：跑 pytest

新建檔案 .github/workflows/test.yml，貼上以下內容、commit、push——下次 push 就會自動跑。

Create .github/workflows/test.yml, paste the snippet below, commit, push — and the next push triggers it automatically.

# .github/workflows/test.yml
name: CI
on: [push, pull_request]
jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-python@v5
        with:
          python-version: '3.11'
      - run: pip install -e ".[test]"
      - run: pytest

逐段解析：

  • name: CI — 在 Actions 頁面顯示的名字。
  • on: [push, pull_request] — 推 code 或開 PR 時觸發。
  • runs-on: ubuntu-latest — 用最新 Ubuntu VM。
  • actions/checkout@v4 — 把 repo 程式碼下載到 runner。
  • actions/setup-python@v5 — 裝 Python 3.11。
  • run: pytest — 執行測試；任何 step 失敗整個 job 就算 fail。

Block-by-block:

  • name: CI — display name in the Actions tab.
  • on: [push, pull_request] — fires on push or PR.
  • runs-on: ubuntu-latest — uses the latest Ubuntu VM.
  • actions/checkout@v4 — checks out the repo onto the runner.
  • actions/setup-python@v5 — installs Python 3.11.
  • run: pytest — runs the tests; any failing step fails the job.

矩陣與密鑰

Matrix builds 與 Secrets

🧮 Matrix：跨版本平行測試

strategy:
  matrix:
    python: ['3.9','3.10','3.11','3.12']
    os: [ubuntu-latest, macos-latest]

4 × 2 = 8 個 job 平行跑。發行 R / Python 套件時拿來驗證跨版本相容性最划算。

4 × 2 = 8 jobs in parallel. Perfect for verifying cross-version compatibility when publishing R/Python packages.

🔐 Secrets：敏感資料怎麼存

步驟：repo Settings → Secrets and variables → Actions → New secret。

workflow 內：${{ secrets.MY_TOKEN }}。每次 run 自帶 ${{ secrets.GITHUB_TOKEN }}，免設定。

最小權限原則：用 permissions: contents: read 鎖死範圍。

How: repo Settings → Secrets and variables → Actions → New secret.

In the workflow: ${{ secrets.MY_TOKEN }}. Every run gets ${{ secrets.GITHUB_TOKEN }} automatically.

Least-privilege principle: scope tightly with permissions: contents: read.

⚠️
地雷：不要把 secrets 放在 workflow 層級的 env:——這會讓所有 step 都看得到。只在需要的 step 裡傳入。 Pitfall: don't put secrets in workflow-level env: — that exposes them to every step. Pass them only into the step that needs them.
真實範例

實戰：scRNA-seq pipeline 每次 PR 自動測試

下面這個 workflow：每次有人對 main 開 PR，就用 1K 個細胞 的測試資料跑一次 Snakemake pipeline。整個 CI 不到 4 分鐘，就能保證 main 永遠跑得起來。

This workflow runs the Snakemake pipeline on a 1K-cell test dataset every time someone opens a PR against main. The whole CI finishes in under 4 minutes — and main is guaranteed runnable.

# .github/workflows/test-pipeline.yml
name: Test scRNA pipeline
on: pull_request
jobs:
  test-pipeline:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: snakemake/snakemake-github-action@v1
        with:
          snakefile: workflow/Snakefile
          args: '--use-conda --cores 2 --config samples=test_samples.tsv'

我的專案需要 CI 嗎？




🌳 What should I CI?




Q1:
要發佈的套件（R / Python）？→ 是 → 用 lint + 多版本 test，tag 時自動 release。


Q2:
可重現 pipeline（Nextflow / Snakemake）？→ 是 → 每個 PR 都用 test profile 跑一次。


Q3:
一篇論文用的單人分析？→ 可選 → 但加 pytest + lint 能在問題變成一週災難前抓到。


Q4:
每週固定的報表？→ 是 → 用 on: schedule cron 每週自動跑。






Q1:
A package you publish (R / Python)? → Yes → Lint, test on multiple versions, deploy on tag.


Q2:
A reproducible pipeline (Nextflow / Snakemake)? → Yes → Run the test profile on every PR.


Q3:
A solo analysis for one paper? → Optional → Adding pytest + lint catches dumb errors before they cost you a week.


Q4:
A scheduled report? → Yes → on: schedule cron to refresh weekly.







💰

費用與額度：公開 repo 免費無限制。私有 repo 免費額度每月 2000 分鐘。Linux = 1× 分鐘、macOS = 10×、Windows = 2×。Self-hosted runner 無限制（但你自己維護）。
Cost & limits: public repos = unlimited free. Private repos = 2000 min/month free. Linux = 1× minute, macOS = 10×, Windows = 2×. Self-hosted runners are unlimited (but you maintain them).

# Python package CI: test + coverage
name: Python CI
on: [push, pull_request]
jobs:
  test:
    runs-on: ubuntu-latest
    strategy:
      matrix:
        python: ['3.10','3.11','3.12']
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-python@v5
        with:
          python-version: ${{ matrix.python }}
      - run: pip install -e ".[test]"
      - run: pytest --cov=src --cov-report=xml
      - uses: codecov/codecov-action@v4
        with:
          token: ${{ secrets.CODECOV_TOKEN }}



# R package CI: R CMD check across versions
name: R-CMD-check
on: [push, pull_request]
jobs:
  check:
    runs-on: ${{ matrix.os }}
    strategy:
      matrix:
        os: [ubuntu-latest, macos-latest]
        r: ['4.3','4.4']
    steps:
      - uses: actions/checkout@v4
      - uses: r-lib/actions/setup-r@v2
        with:
          r-version: ${{ matrix.r }}
      - uses: r-lib/actions/setup-r-dependencies@v2
      - uses: r-lib/actions/check-r-package@v2



# Nextflow / nf-core pipeline test profile
name: nf-core test
on: pull_request
jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: nf-core/setup-nextflow@v1
      - run: nextflow run . -profile test,docker --outdir results
      - uses: actions/upload-artifact@v4
        if: always()
        with:
          name: nextflow-log
          path: .nextflow.log



# Build & push Docker image to GHCR
name: Docker
on:
  push:
    tags: ['v*']
jobs:
  build:
    runs-on: ubuntu-latest
    permissions:
      contents: read
      packages: write
    steps:
      - uses: actions/checkout@v4
      - uses: docker/setup-buildx-action@v3
      - uses: docker/login-action@v3
        with:
          registry: ghcr.io
          username: ${{ github.actor }}
          password: ${{ secrets.GITHUB_TOKEN }}
      - uses: docker/build-push-action@v5
        with:
          push: true
          tags: ghcr.io/${{ github.repository }}:${{ github.ref_name }}
