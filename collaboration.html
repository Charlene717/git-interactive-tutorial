STEP 8 / 9

GitHub 協作 (PR & Review)

Git 只是版本控制，GitHub 把它變成一個社群——Issue、Pull Request、Code Review 是讓多人協作不互相覆寫的關鍵。

Git is just version control; GitHub layers collaboration on top — issues, pull requests, and code review are what let many people work on one codebase without trampling each other.

概覽

從版本控制到社群協作

Git 本身只解決一件事：誰、在何時、改了什麼。GitHub 在這之上加了一整層協作介面——Issue 追蹤、Pull Request 審查、Project 看板、Organization 權限——讓開源專案（如 Seurat、Scanpy、nf-core）能由數百人共同維護。

本章的核心工作流可以濃縮成一條鏈：fork → branch → push → PR → review → merge。後面每一節都是在拆解這條鏈的某一段。

Git answers one question: who changed what, when. GitHub layers an entire collaboration UI on top — issue tracking, pull-request review, project boards, organization permissions — which is how open-source projects like Seurat, Scanpy, and nf-core get maintained by hundreds of contributors.

This chapter compresses to one chain: fork → branch → push → PR → review → merge. Each section below unpacks one link in that chain.

💡
核心心智模型：PR 不是「提交程式碼」，而是「請求把我的分支合進去」——它是一個 對話框，包含 diff、討論、CI 結果、審查意見。 Mental model: a PR isn't "submitting code" — it's "requesting that my branch be merged in." It's a conversation thread wrapped around a diff, with CI results and reviewer comments attached.
FORK

Fork 與 Clone 的差別

Fork 是在 GitHub 伺服器上把整個 repo 複製到 你的帳號 底下（你擁有寫入權）；Clone 則是把 repo 下載到 你的本機。兩者不互斥——典型開源貢獻是「先 fork，再 clone fork」。

A fork is a server-side copy of a repo under your account (you own write access to it). A clone is a local copy on your machine. They're not mutually exclusive — typical open-source contribution is "fork first, then clone the fork."

🍴 何時 fork？

你 沒有寫入權 的 repo——大部分開源工具（Seurat、Scanpy、nf-core/rnaseq）都屬於此類。Fork 後你在自己副本內推進，再透過 PR 把改動送回 upstream。

You don't have write access — true for most OSS tools (Seurat, Scanpy, nf-core/rnaseq). You push into your own copy, then open a PR back to upstream.

📥 何時只 clone？

你 有寫入權——個人 repo、實驗室 organization 下的專案。直接 clone、開 feature branch、推回 origin、開 PR 就好。

You have write access — your own repos, your lab's org repos. Just clone, branch, push to origin, and open a PR there.

🔺
「三角工作流」：upstream（原作者的 canonical repo）↔ origin（你 fork 在 GitHub 的副本）↔ local（你的本機）。本機 commit 後 git push origin，再到 GitHub 從 fork 開 PR 到 upstream。記得定期 git pull upstream main 同步。 The "triangle workflow": upstream (the canonical repo) ↔ origin (your fork on GitHub) ↔ local (your machine). Commit locally, git push origin, then open a PR from your fork to upstream on the GitHub UI. Periodically git pull upstream main to stay in sync.
PR

一個 Pull Request 長什麼樣？

一個 PR 在 GitHub UI 上分成 4 個 tab，加上「標題＋描述」這個核心區。理解每一塊在做什麼，才能寫出評審員看了願意合進去的 PR。

A PR has 4 tabs on GitHub plus the title-and-description core. Knowing what each part is for is the difference between a PR that gets merged and one that stalls.

區塊內容最佳實踐
Title一句話總結改動祈使句、簡短：Add SCTransform v2 normalization optionOne-line summary of the changeImperative, concise: Add SCTransform v2 normalization option
DescriptionWhat / Why / 截圖 / 測試計畫用 Fixes #42 連結 issue，合進去時自動關閉What / why / screenshots / test planLink issues with Fixes #42 — auto-closes them on merge
Commits tab這次 PR 推上來的所有 commitcommit message 寫清楚，方便 squash 時參考Every commit pushed for this PRWrite clear messages — they help when squashing
Files Changed tab完整 diff，可在行上留 inline commentPR 控制在 < 400 行有效更動Full diff with inline-comment supportKeep effective changes under ~400 lines
Conversation tab討論、approval、change request每個 review comment 都要回覆Discussion, approvals, change-requestsReply to every reviewer comment
Checks tabCI 結果（GitHub Actions、Codecov…）紅燈不要請 review——詳見 Advanced 2CI results (Actions, Codecov, …)Don't request review while red — see Advanced 2
DRAFT

Draft PR 與 Ready PR

新功能還沒寫完？開 Draft PR——CI 會跑、團隊看得到方向、但不會被當成「請審查」。寫完了再點「Ready for review」轉成正式 PR。事後想暫停評審也可以「Convert to draft」收回。

對 bioinformatics 專案特別實用：分析腳本很長、想先把 plot 截圖貼上來確認方向，draft PR 是最低摩擦的辦法。

Feature still in progress? Open a Draft PR — CI runs, teammates can see the direction, but reviewers aren't pinged. When you're done, click Ready for review to promote it. You can also Convert to draft later to pause review.

Especially useful for bioinformatics: analyses can sprawl, and posting a plot screenshot inside a draft PR is the lowest-friction way to ask "is this the right direction?"

REVIEW

Code review 禮儀

🔍 當評審員 (Reviewer)

•具體：「這裡 NaN 沒處理」優於「程式碼很怪」。
•提替代方案：別只罵，附上能跑的範例。
•區分 blocking（一定要改）和 nit（小意見）。
•別 bikeshed——空格、命名爭一週不值得。
•覺得 OK 就按 Approve，別卡著。

•Be specific: "NaN not handled here" beats "this is weird."
•Suggest an alternative: don't just criticize — show a working snippet.
•Mark issues as blocking vs nit.
•Don't bikeshed — a week-long debate over spacing isn't worth it.
•If it's good, hit Approve — don't sit on it.

✍️ 當作者 (Author)

•每個 comment 都回覆，即使只是 Done.
•修正時推 新 commit——別 git push --force，會把評審員的 anchor 沖掉。
•合進去前再 squash 不遲。
•謝謝評審員——他們的時間是免費的禮物。

•Reply to every comment, even if just Done.
•Push new commits for fixes — don't git push --force mid-review; it wipes the reviewer's line anchors.
•Squash later, before merge.
•Thank your reviewers — their time is a free gift.

✨
Suggested changes：評審員可以在 inline comment 內用 ```suggestion 區塊提出修改，作者一鍵點 Commit suggestion 就能採納。打字一行的成本，比解釋十行還省。 GitHub's "Suggested changes": reviewers can wrap an edit in a ```suggestion block inside an inline comment; the author accepts with one click. Typing the one-line fix is often cheaper than explaining it.
ISSUES

Issue、Label、Assignee

Issue 是 GitHub 的 bug/feature/discussion 追蹤板——任何「待辦」都該先開一個 issue，再用 PR 解決它。三個維度幫你整理：

  • Labels：bug、enhancement、good first issue、help wanted——分類、過濾、新手友善。
  • Assignees：指派負責人；沒人 assign 的 issue 通常會被遺忘。
  • Milestone：對應版本（如 v2.0 release）。

在 PR 描述寫 Fixes #42（或 Closes、Resolves），PR 被 merge 那一刻 issue #42 會自動關閉——少一步手動清單。

Issues are GitHub's bug/feature/discussion tracker — anything you owe the codebase should start as an issue, then get resolved by a PR. Three knobs:

  • Labels: bug, enhancement, good first issue, help wanted — for filtering and onboarding newcomers.
  • Assignees: who owns it; unassigned issues tend to evaporate.
  • Milestone: groups issues by release (v2.0 release).

Write Fixes #42 (or Closes, Resolves) in the PR description, and issue #42 auto-closes the moment the PR is merged — one fewer step in your housekeeping.

MERGE

GitHub 上的三種 merge 策略

策略實際發生什麼何時用
Create merge commit保留分支形狀，新增一個 merge commit有意義的 feature branch、想保留協作軌跡Preserves the branch shape, adds a merge commitMeaningful feature branches; you want collaboration history visible
Squash and merge把 PR 內所有 commit 壓成 1 個PR 內有很多 WIP commit、想要 main 線乾淨Collapses every PR commit into oneMany WIP commits in the PR; you want main clean
Rebase and merge把 PR commit 一個個重播到 base 上，無 merge commitcommit 已經很乾淨、想要線性歷史Replays PR commits linearly onto base, no merge commitCommits are already clean; you want a linear history
⚠️
共識：每個專案應該在 README 或 CONTRIBUTING.md 中寫清楚要用哪一種——團隊混用最痛苦。Scanpy 用 squash，nf-core 用 merge commit，Linux kernel 用 rebase。 Convention: each project should pick one strategy and document it in README or CONTRIBUTING.md — mixing is the painful path. Scanpy uses squash, nf-core uses merge commits, the Linux kernel uses rebase.

完整流程

完整 PR 工作流程

# 1. 在 GitHub 網頁右上角點「Fork」按鈕 / Click "Fork" on the upstream repo page
#    => 你的帳號底下會出現 github.com/charlene717/scanpy

# 2. Clone 你的 fork（不是原作者的 repo） / Clone YOUR fork
$ git clone git@github.com:charlene717/scanpy.git
$ cd scanpy

# 3. 加入 upstream remote，方便日後同步原作者更新
$ git remote add upstream git@github.com:scverse/scanpy.git
$ git remote -v
origin    git@github.com:charlene717/scanpy.git (push)
upstream  git@github.com:scverse/scanpy.git     (fetch)

# 4. 開 feature branch / Branch off main
$ git switch -c fix/sctransform-nan-handling

# 5. 編輯、commit / Edit and commit
$ git add scanpy/preprocessing/_normalization.py
$ git commit -m "Handle NaN values in SCTransform v2"

# 6. Push 到你的 fork / Push to YOUR fork
$ git push -u origin fix/sctransform-nan-handling

# 7. 到 GitHub 點「Compare & pull request」開 PR
#    Base: scverse/scanpy:main  ←  Compare: charlene717/scanpy:fix/sctransform-nan-handling
#    填標題、描述（含 Fixes #1234）、提交！
# 你的實驗室 organization 下、你有寫入權的 repo
# Your lab's org repo where you have push access

# 1. Clone（直接 clone 原 repo，不用 fork） / Clone directly (no fork needed)
$ git clone git@github.com:my-lab/scRNA-pipeline.git
$ cd scRNA-pipeline

# 2. 開 feature branch / Branch off main
$ git switch -c feature/add-doublet-filter

# 3. 工作 + commit / Work and commit
$ git add R/doublet.R tests/test-doublet.R
$ git commit -m "Add scDblFinder integration for doublet removal"

# 4. Push 到同一個 repo / Push to the same repo
$ git push -u origin feature/add-doublet-filter

# 5. 開 PR：base = main ← compare = feature/add-doublet-filter
#    Request review from @PI, @senior-postdoc
# 用 GitHub CLI（gh）一條龍 / Doing it all with GitHub CLI
$ gh auth login                                       # 首次登入

# 開 PR（自動偵測 branch 和 base）
$ gh pr create --title "Fix SCTransform NaN handling" \
               --body  "Fixes #1234. Replaces NaN with row median." \
               --base  main

# 列出當前 repo 的開放 PR
$ gh pr list
#142  Fix SCTransform NaN handling   charlene717:fix/sctransform-nan-handling

# 拉某個 PR 下來 review（自動切到對應 branch）
$ gh pr checkout 142

# 直接在 CLI 評論 / 批准 / 要求修改
$ gh pr review 142 --approve -b "LGTM, thanks!"
$ gh pr review 142 --request-changes -b "Need test coverage"

# Merge（squash / rebase / merge 三選一）
$ gh pr merge 142 --squash --delete-branch
生資現場

生物資訊實務：PR 與你的論文

📐
小 PR 比較快 merge：研究顯示 < 400 行有效更動 的 PR 比起千行巨獸，merge 速度快 3 倍，被找到 bug 的機率也高 2 倍。把大功能拆成 3–5 個 PR 是值得的紀律。 Small PRs merge faster: studies show PRs under ~400 lines changed get merged 3× faster than thousand-line monsters, and reviewers catch 2× more bugs. Splitting a big feature into 3–5 PRs is a discipline that pays off.
🚫
別自己 merge 自己的 PR：就算技術上你能按下 Merge，在共用專案請別這麼做。PR 的全部意義就是「第二雙眼睛」——這個門檻不該為了快而省。 Don't merge your own PR without review: even if you technically can. On a shared project, the whole point of a PR is the second pair of eyes — don't trade that away for speed.
📄
論文裡引用你的 PR：在 Methods 或 Code availability 段落貼 PR URL，例如 https://github.com/lab/pipeline/pull/42。Reviewer 想看就點得到，且 commit hash 永久存在。 Cite your PR in the paper: drop the URL into Methods or Code Availability, e.g. https://github.com/lab/pipeline/pull/42. Reviewers can inspect the exact change, and the commit hash is permanent.
