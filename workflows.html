Git 工作流程 (Git Workflows)

同樣的 Git 指令，可以編出完全不同的舞步——選對工作流程，整個團隊都會輕鬆。

The same Git commands can be choreographed in very different ways — picking the right workflow saves arguments and reduces merge pain.

概覽

什麼是「工作流程」？

「工作流程」其實是一種團隊約定——大家如何使用分支、什麼時候 merge、怎麼發 release。底層的 Git 指令都一樣（branch / merge / tag），差別在編舞：分支怎麼命名？誰能 push 到 main？release 從哪條分支切？

選對工作流程，可以省下無數次「為什麼你直接 push 到 main？」的爭執，並大幅降低 merge 衝突。本頁介紹三種主流模式、語意化版本與 conventional commits，再給你一個「工作流程推薦器」。

A “workflow” is really a team convention — how you all agree to use branches, when to merge, how to ship releases. The underlying Git commands (branch / merge / tag) are identical across workflows; what differs is the choreography: branch naming, who can push to main, where releases get cut.

Picking the right one saves countless “why did you push straight to main?” arguments and reduces merge pain. This page covers three mainstream models, semantic versioning, and conventional commits — plus an interactive recommender at the end.

💡
核心提醒：對 95% 的生資實驗室專案來說，GitHub Flow + semantic versioning 已經夠用。不要過度工程。 Key takeaway: for 95% of bioinformatics lab projects, GitHub Flow + semantic versioning is enough. Don’t over-engineer.
GITHUB FLOW

一、GitHub Flow（最簡單）

只有一條長期分支 main，永遠保持可部署狀態。每個改動都是短命分支：從 main 切 → commit → push → 開 PR → review → merge → 刪分支。

  • 優點：簡單、易懂、適合持續部署。
  • 缺點：無法同時維護多個已釋出版本。
  • 最適合：網頁 app、單一版本工具、絕大多數的實驗室專案。

One long-lived branch — main — always deployable. Every change is a short-lived branch: branch from main → commit → push → open PR → review → merge → delete the branch.

  • Pros: simple, easy to teach, great for continuous deployment.
  • Cons: can’t maintain multiple released versions in parallel.
  • Best for: web apps, single-version tools, and most lab projects.
# GitHub Flow 典型循環 / Typical GitHub Flow cycle
$ git switch -c feature/sctransform-v2 main
$ # ... 編輯、測試 ...
$ git commit -am "feat: add SCTransform v2 option"
$ git push -u origin feature/sctransform-v2
# 在 GitHub 開 PR → review → squash & merge → delete branch
GIT FLOW

二、Git Flow（重量級）

由 Vincent Driessen 在 2010 年提出。兩條長期分支：

  • main——只放已釋出的正式版（每個 commit 都對應一個 tag）。
  • develop——日常開發整合分支。

加上三類短命分支：feature/*（從 develop 切）、release/*（準備發版）、hotfix/*（修 production bug，從 main 切）。

  • 優點：支援多版本平行維護。
  • 缺點：複雜，2025 年多數團隊覺得殺雞用牛刀。

有趣的是，連 Driessen 本人現在都建議：網頁/SaaS 類專案改用 GitHub Flow 就好。

Popularized by Vincent Driessen in 2010. Two long-lived branches:

  • main — only tagged production releases live here.
  • develop — daily integration branch.

Plus three short-lived branch types: feature/* (cut from develop), release/* (preparing a version), hotfix/* (fixing a production bug, cut from main).

  • Pros: supports maintaining multiple released versions in parallel.
  • Cons: complex; in 2025 many teams find it overkill.

Notably, even Driessen now recommends GitHub Flow for web / SaaS projects.

TRUNK-BASED

三、Trunk-based 開發

所有人直接 commit 到 main（用 feature flags 隱藏未完成功能），或只開不到一天的超短命分支。需要重度 CI 把關。

  • 優點：速度最快、沒有累積的「整合債」。
  • 缺點：需要紀律 + 完整自動化測試；對 ad-hoc 科研工作不適合。

Everyone commits to main directly (using feature flags to hide unfinished work), or via branches that live less than a day. Requires heavy CI.

  • Pros: highest velocity, no accumulated integration debt.
  • Cons: demands discipline plus full test automation; not ideal for ad-hoc science work.

SEMVER

五、語意化版本與 tag

軟體版本長得像 v1.4.2，格式為 MAJOR.MINOR.PATCH：

  • MAJOR：破壞性更新（API 不相容）
  • MINOR：新增功能、向下相容
  • PATCH：bug fix

Versions look like v1.4.2 in the form MAJOR.MINOR.PATCH:

  • MAJOR: breaking change (API incompatible)
  • MINOR: new feature, backwards-compatible
  • PATCH: bug fix
# 建立 annotated tag / Create an annotated tag
$ git tag -a v1.4.2 -m "Release 1.4.2 — fix NA handling in QC"
$ git push --tags

# 直接在 GitHub 建 release（含 release notes + binaries）
$ gh release create v1.4.2 --notes-from-tag

GitHub Releases 把 tag + release notes + 附件（binaries / 範例資料）綁在一起，使用者可以 pin 到特定版本，paper 也能引用「v1.4.2」這個確切的快照。

A GitHub Release bundles a tag + release notes + attached assets (binaries, example data). Users can pin to a specific version, and papers can cite the exact snapshot “v1.4.2.”

LAB CASE STUDIES

六、實驗室常見場景該選哪個

📄 一次性論文專案

GitHub Flow。投稿時切一個 release/manuscript-v1 凍結快照；被接受後在 main 打 tag manuscript-submitted。後續 revision 直接回 main 開 PR。

GitHub Flow. Cut a release/manuscript-v1 snapshot at submission, then tag main as manuscript-submitted upon acceptance. Future revisions land on main via PR.

🧪 實驗室核心 pipeline

GitHub Flow + semver tag。每個版本 tag 一個 v1.x.x，組員可以 pin 到指定版本跑分析，避免 main 改動造成「重跑結果不一致」。

GitHub Flow with semver tags. Tag each version (v1.x.x) so lab members can pin to a specific release and avoid “the result changed because main moved.”

🤝 維護開源工具

Forking workflow + semver + release notes + CHANGELOG.md。外部貢獻者 fork 後開 PR；維護者控制 main，定期打 tag。

Forking workflow + semver + release notes + CHANGELOG.md. External contributors fork and open PRs; maintainers gate main and tag releases periodically.

👤 單兵作戰的博班生

最簡單：直接 commit 到 main，投稿時打一個 tag。沒必要開分支，除非要做高風險實驗（這時再開 experiment/* 分支）。

Simplest: commit straight to main and tag at submission. No need for branches unless you’re running a risky experiment — then a one-off experiment/* branch is fine.

DECISION TREE

八、「我該從哪條分支切？」決策樹

🌳 分支起點決策樹

Q1:
要做新功能？→ 從 main（GitHub Flow）或 develop（Git Flow）切。
Q2:
要修 production bug？→ 從 main 切 hotfix/* 分支，修完合回 main 並打 tag。
Q3:
要準備發版？→ 僅 Git Flow：從 develop 切 release/vX.Y.Z，凍結功能、修最後 bug 後合進 main + develop。
Q1:
Building a new feature? → Branch from main (GitHub Flow) or develop (Git Flow).
Q2:
Fixing a production bug? → Cut a hotfix/* branch from main, fix, merge back, and tag.
Q3:
Preparing a release? → Git Flow only: cut release/vX.Y.Z from develop, freeze features, polish, then merge into both main and develop.
COMMIT MSG

九、Commit message 慣例

Conventional Commits 是一種社群約定的訊息格式：<type>: <summary>。常見 type：

Conventional Commits is a community message format: <type>: <summary>. Common types:

feat:     add SCTransform v2 option        # 新功能 / new feature
fix:      handle NA in MT% filter          # bug fix
docs:     update README with citation      # 文件 / docs only
refactor: extract QC into module           # 結構重構，行為不變
test:     add unit tests for normalization # 新增測試
chore:    bump R version to 4.4            # 雜事 / housekeeping
feat!:    change DE function signature     # ! = BREAKING CHANGE

讀得懂這格式的工具：semantic-release（自動 bump 版本）、commitizen（互動式輸入 commit）、各種 changelog generator。

Tools that read this format: semantic-release (auto-bumps versions), commitizen (interactive commit prompts), and various changelog generators.

完整程式碼

十、實際操作範例

# GitHub Flow — 完整循環 / End-to-end cycle
$ git switch main && git pull
$ git switch -c feature/qc-filter-tweak
# ... 修改檔案 ...
$ git commit -am "feat: tighten MT% threshold to 15"
$ git push -u origin feature/qc-filter-tweak

# 在 GitHub 開 PR：
$ gh pr create --fill --reviewer @teammate

# review 通過後 squash & merge，然後刪分支
$ gh pr merge --squash --delete-branch
# 使用 git flow CLI extension
$ brew install git-flow-avh         # macOS
$ sudo apt install git-flow         # Ubuntu

$ git flow init -d                  # 接受預設（main + develop）

# 開新 feature（從 develop 切）
$ git flow feature start sctransform-v2
$ # ... 編輯、commit ...
$ git flow feature finish sctransform-v2   # 合回 develop 並刪分支

# 準備發版
$ git flow release start 1.4.0
$ # ... 修 release-blocker、改 version 字串 ...
$ git flow release finish 1.4.0     # 合進 main + develop + 自動 tag
$ git push --all && git push --tags
# 1. 建立 annotated tag（推薦，含作者與日期）
$ git tag -a v1.4.2 -m "Release 1.4.2 — fix NA in QC"

# 2. 推送 tag 到遠端（注意：push 預設不會帶 tag）
$ git push origin v1.4.2
# 或一次推全部 tag：
$ git push --tags

# 3. 在 GitHub 建 release（從 tag 自動帶 notes）
$ gh release create v1.4.2 --notes-from-tag

# 4. 附加二進位檔 / Attach binaries
$ gh release upload v1.4.2 ./dist/my-tool-v1.4.2.tar.gz

# 5. 查看所有 tag
$ git tag --list "v*" --sort=-v:refname | head
⚠️
禁忌：不要在專案進行到一半時，未經團隊同意就切換工作流程。「半套 Git Flow」比「沒 Git Flow」還糟糕——大家會搞不清楚該往哪條分支 push。 Don’t do this: never switch workflows mid-project without team agreement. Half-applied Git Flow is worse than no Git Flow — nobody will know which branch to push to.
