檢視變更 (Inspecting Changes)

Git 是一台時光機；這一章告訴你怎麼轉動上面的旋鈕——看清楚每一次 commit 改了什麼、是誰、又是何時。

Git is a time machine; this chapter shows you the dials — see exactly what each commit changed, who made the change, and when.

概覽

為什麼要學「檢視」？

當你已經做了幾十個 commit，問題就會變成：「上週那段 p_val_adj 過濾條件是誰改的？」「某個 script 是哪一個 commit 弄壞的？」「現在 staging 區到底要 commit 什麼？」這些都不能靠記憶解決——必須用 Git 內建的檢視工具。

本章涵蓋五個核心指令：git diff、git log、git show、git blame、git reflog。它們是 bug hunting、論文重現、code review 的瑞士刀。學會以後，你能精準指出「Figure 3 是 commit a3f9b21 生成的」——這正是可重現研究的核心。

After a few dozen commits, the questions become: "Who last changed the p_val_adj threshold?" "Which commit broke analysis.py?" "What exactly will I be committing right now?" You can't answer these from memory — you need Git's inspection tools.

This chapter covers five core commands: git diff, git log, git show, git blame, and git reflog. They are the Swiss army knife for bug hunting, paper reproduction, and code review. Once mastered, you can point at a specific commit and say "Figure 3 was generated by a3f9b21" — which is exactly what reproducibility means.

🕰️
把 Git 想成時光機：每個 commit 是一個時間切片，每個指令是不同的「看時間」方式。log 看時間線，diff 比較兩個切片，show 放大某一個切片，blame 反查每一行屬於哪個切片。 Think of Git as a time machine: each commit is a snapshot, each command is a different way of viewing time. log shows the timeline, diff compares two snapshots, show zooms into one snapshot, blame traces each line back to its snapshot.
三棵樹

先建立三區心智模型

Git 同時維護三個「樹」：Working Directory（你正在編輯的檔案）、Index / Staging（已 git add 但還沒 commit）、HEAD（最後一次 commit 的快照）。所有 diff 指令都在比較其中兩棵樹。

Git tracks three trees simultaneously: Working Directory (files you're editing), Index / Staging (already git added but not committed), and HEAD (the last commit's snapshot). Every diff command compares two of these.

📝

git diff

Working ↔ Index

還沒 add 的變更。回答：「我接下來要 add 什麼？」

Changes not yet added. Answers: "What am I about to stage?"

📦

git diff --staged

Index ↔ HEAD

已 add 但尚未 commit 的變更。回答：「這次 commit 會包含哪些變更？」

Staged but not yet committed. Answers: "What will be in my next commit?"

🌐

git diff HEAD

Working ↔ HEAD

所有還沒 commit 的變更（staged + unstaged 合計）。

All uncommitted changes (staged + unstaged combined).

# 視覺化：三棵樹彼此的關係 / Visualize the three trees
   Working Directory  ──[ git add ]──▶  Index (Staging)  ──[ git commit ]──▶  HEAD
        ▲                                      ▲                                ▲
        └────────────── git diff ──────────────┘                                │
                                               └──── git diff --staged ─────────┘
        └────────────────────────── git diff HEAD ──────────────────────────────┘
log 指令

一、git log 進階用法

裸 git log 會印出長長的提交歷史。透過 flag 組合，你能把它變成精準的查詢工具。

Bare git log dumps a long commit history. With the right flags, it becomes a precise query tool.

diff 細節

二、git diff 與 unified diff 格式

Git 預設輸出 unified diff。先看一個例子，再逐行拆解：

Git outputs unified diff format by default. Here's an example, dissected line-by-line:

diff --git a/analysis.R b/analysis.R           ← 兩個版本的檔案 / two file versions
index 9f3c1a..a7b2e4 100644                    ← blob hash + mode
--- a/analysis.R                                ← 舊版（"a") / old version
+++ b/analysis.R                                ← 新版（"b") / new version
@@ -12,5 +12,7 @@ run_de(seurat_obj)         ← 老檔第 12 行起 5 行；新檔 12 行起 7 行
   markers <- FindMarkers(obj,
                          ident.1 = "T_cell",
-                          logfc.threshold = 0.25,
-                          min.pct = 0.1)
+                          logfc.threshold = 0.5,
+                          min.pct = 0.25,
+                          test.use = "MAST")
   markers %>% filter(p_val_adj < 0.05)

關鍵符號：

  • @@ -12,5 +12,7 @@ = 「hunk 標頭」：舊檔自第 12 行起 5 行 → 新檔自第 12 行起 7 行
  • - 開頭 = 舊版有但新版沒有（被刪掉）
  • + 開頭 = 新版新增
  • 沒前綴 = 上下文（context lines，沒改）

Key symbols:

  • @@ -12,5 +12,7 @@ is the "hunk header": old file 5 lines from line 12 → new file 7 lines from line 12
  • - prefix = present in old, gone in new (deleted)
  • + prefix = new in this version
  • No prefix = context line (unchanged)
# 常用 diff flag / Common diff flags
$ git diff --stat                  # 只看檔案統計 / files-changed summary
$ git diff --word-diff             # word-level diff（適合 markdown / 論文稿）
$ git diff HEAD~3..HEAD            # 比較 3 個 commit 前 vs 現在
$ git diff main..feature/qc        # 比較兩個分支
$ git diff main..feature -- src/   # 只看 src/ 目錄的差異
blame & show

三、git blame 與 git show

🔎 git blame

逐行標示「誰、在哪個 commit、於何時最後修改了這一行」。Debug 時的第一步：「這個被改掉的參數是誰動的？」

Annotates each line with "who, in which commit, when last changed this line." First step in debugging: "Who touched this parameter?"

$ git blame analysis.R
a3f9b21 (Alice  2026-05-10) markers <- FindMarkers(obj,
b7c4e82 (Bob    2026-05-08)   logfc.threshold = 0.5,
b7c4e82 (Bob    2026-05-08)   test.use = "MAST")
9c0a55b (Alice  2026-04-21) ggplot(markers, aes(...))

🔬 git show

顯示單一 commit 的完整資訊：作者、時間、message、所有檔案的 diff。也能用來「不 checkout」直接看某個歷史版本的檔案內容。

Shows full info for a single commit: author, time, message, and diffs for every file. Can also view a file as of a specific commit without checking it out.

$ git show a3f9b21              # 完整 commit / full commit
$ git show a3f9b21 --stat        # 只看統計 / stats only
$ git show a3f9b21:analysis.R    # 該 commit 當時的檔案內容
🧪
實戰流程：看到 bug → git blame <file> 找出 hash → git show <hash> 看當時為什麼這樣改 → 讀 commit message 通常就有答案。如果 message 寫得模糊，那是另一個議題（請見 commit message 章節）。 Practical workflow: spot a bug → git blame <file> for the SHA → git show <sha> for context → the commit message often explains why. If the message is too vague, that's a different problem (see the commit-message chapter).

實用程式碼


實戰指令組合



  
  
  




# 一天工作開始時的標準三連 / Standard morning routine
$ git status                       # 現在工作區乾不乾淨？
$ git diff                         # 還沒 add 的變更
$ git diff --staged                # 已 add 等待 commit 的變更

# 看最近 10 個 commit 摘要 / Recent 10 commits at a glance
$ git log --oneline -10

# 看自己的 commit / Your own commits this month
$ git log --author="$(git config user.name)" --since="1 month ago" --oneline

# 看某個檔案的完整歷史 / Full history of one file
$ git log --follow -p -- analysis.R




# 場景：analysis.py line 47 出 bug，要找元兇
# Scenario: bug on analysis.py line 47, hunt the culprit

# 1) blame 拿到 SHA / Step 1: blame to get the SHA
$ git blame -L 40,50 analysis.py

# 2) show 看當時為什麼這樣改 / Step 2: show why it was changed
$ git show <sha>

# 3) 如果是字串被改了，用 pickaxe / If a string changed, use pickaxe
$ git log -S "p_val_adj < 0.05" --oneline -- analysis.py

# 4) 二分搜尋找哪個 commit 開始壞掉（進階）/ Binary-search broken commit (advanced)
$ git bisect start
$ git bisect bad                   # 現在這個版本是壞的
$ git bisect good v1.0             # v1.0 是好的
# Git 會自動 checkout 中間的 commit，你跑測試後告訴它 good/bad
$ git bisect reset                 # 結束 / End




# 一個能用一輩子的 log alias / A log alias for life
$ git config --global alias.lg "log --graph --decorate --all \
  --pretty=format:'%C(yellow)%h%Creset %C(cyan)%an%Creset %C(green)(%ar)%Creset%C(red)%d%Creset %s' \
  --abbrev-commit"

# 之後輸入 / Then just type:
$ git lg
* a3f9b21 Alice (2 hours ago) (HEAD -> main) fix: correct p_val_adj filter
* b7c4e82 Bob (yesterday) feat: add Seurat clustering
| * 2e1d9f4 Alice (2 days ago) (feature/qc) refactor QC
|/
* 9c0a55b Alice (1 week ago) (tag: v0.1) initial DE script

# 其他常用 alias / Other handy aliases
$ git config --global alias.last "log -1 HEAD --stat"
$ git config --global alias.who  "shortlog -sn --all --no-merges"
$ git config --global alias.df   "diff --word-diff"











生資現場


生物資訊實驗室場景：追蹤閾值的歷史


🧬


情境：論文 reviewer 問：「為什麼 DE 分析用 p_val_adj < 0.05？最初的設計是 < 0.01 嗎？」這時你不用翻 lab notebook：


$ git log -S "p_val_adj" --oneline -- DE_analysis.R
a3f9b21 fix: relax p_val_adj from 0.01 to 0.05 (per reviewer #2)
9c0a55b initial DE script

$ git show a3f9b21
# commit 訊息會清楚說明：「Reviewer 2 建議放寬以納入更多基因」


這就是版本控制在研究中的真正價值——可重現性意味著你能精準指出「Figure 3 是 commit a3f9b21 生成的，閾值從那個 commit 起被改為 0.05」。沒有 Git，這種對話就只能靠記憶與 lab notebook，而兩者都會出錯。


Scenario: A reviewer asks: "Why does the DE analysis use p_val_adj < 0.05? Wasn't the original cutoff < 0.01?" You don't need to dig through your lab notebook:


$ git log -S "p_val_adj" --oneline -- DE_analysis.R
a3f9b21 fix: relax p_val_adj from 0.01 to 0.05 (per reviewer #2)
9c0a55b initial DE script

$ git show a3f9b21
# The commit message explains: "Reviewer 2 asked for a looser cutoff to include more genes"


This is the real value of version control in research — reproducibility means being able to say "Figure 3 was generated by commit a3f9b21, and the threshold has been 0.05 since that commit." Without Git, this conversation depends on memory and a lab notebook, both of which fail you.





⚠️

注意：git blame 顯示的是「最後一次修改該行的 commit」——如果有人後來只是 reformat 程式碼（例如自動 lint），那個 commit 就會蓋過真正改邏輯的 commit。遇到這種狀況，加上 git blame -w（忽略空白）或 -C（追蹤檔案間的移動）會更精準。
Gotcha: git blame shows "the commit that last modified each line" — so a later reformat (e.g., an auto-linter) can mask the real logic change. Use git blame -w (ignore whitespace) or -C (track code moved across files) for more accurate attribution.
