撤銷與時光旅行 (Undoing & Time Travel)

Git 最大的超能力是「安全撤銷」——幾乎所有東西都可以救回來。本章教你看清「失誤在哪一層」，然後選對工具反悔。

Git's biggest superpower is "safe undo" — almost everything is recoverable. This chapter teaches you to spot where the mistake lives and reach for the right tool.

概覽

為什麼需要學「撤銷」？

真實的分析工作很少是直線前進：你把 differential_expression.R 改壞了、誤刪了 QC_plots/ 資料夾、把開發用的 API_KEY commit 進去、甚至剛剛 reset --hard 把昨天整天的工作清乾淨——這些都是真實會發生的事。

好消息：Git 把幾乎每一個 HEAD 移動都記錄下來，保存 90 天。學會撤銷的真正關鍵，不是背指令，而是判斷失誤在哪一層：工作區（working dir）？暫存區（index）？已 commit 但沒 push？還是已經推到 GitHub 給別人 pull 走了？層級不同，工具就不同。

Real analysis work is rarely linear: you break differential_expression.R, accidentally delete the QC_plots/ folder, commit an API_KEY, or even reset --hard away yesterday's entire workday. These things happen.

The good news: Git logs almost every HEAD move and keeps it for 90 days. The real key to undoing isn't memorizing commands — it's identifying which layer the mistake lives in: working dir? staging index? committed but unpushed? already pushed and pulled by collaborators? Different layers need different tools.

💡
核心心法：「先別 panic、先想層級」。99% 的災難都能用 git reflog 找回 SHA，再 git checkout 救回來。 Mantra: "Don't panic, identify the layer." 99% of disasters can be recovered by finding the SHA in git reflog and then git checkout-ing it.
決策流程

撤銷決策樹：5 個問題挑工具

遇到失誤時，依序問自己這 5 題，就會知道用哪一個指令。

When something goes wrong, ask yourself these 5 questions in order — the answer tells you which command to use.

🌳 撤銷工具決策流程

Q1:
改壞了某個檔案，還沒 git add？→ 是 → git restore <file>（舊版：git checkout -- <file>），把工作區還原成上次 commit 的版本。
Q2:
已經 add 到暫存區，但還沒 commit？→ 是 → git restore --staged <file>（把它從暫存區拿出來，但保留工作區編輯）。
Q3:
已經 commit、但還沒 push？→ 是 → 三選一：reset --soft HEAD~1（commit 退回、變更保留在暫存區）／ reset --mixed HEAD~1（預設、變更回到工作區）／ commit --amend（修正最後一次 commit 的訊息或內容）。
Q4:
已經 push 到 GitHub？→ 是 → git revert <sha>。它會產生一個「反向 commit」，安全又保留歷史，絕對不要在共用分支用 reset。
Q5:
東西不見了（誤刪 branch、誤 reset --hard）？→ 是 → git reflog 找到那個 dangling SHA，然後 git checkout <sha> 或 git branch recovered <sha>。
Q1:
Modified a file but haven't run git add yet? → Yes → git restore <file> (older syntax: git checkout -- <file>) restores the working copy to the last commit.
Q2:
Already add-ed, not yet committed? → Yes → git restore --staged <file> unstages it but keeps your edits in the working dir.
Q3:
Already committed, not yet pushed? → Yes → Pick one: reset --soft HEAD~1 (rolls back commit, keeps changes staged) / reset --mixed HEAD~1 (default, changes back in working dir) / commit --amend (fix last commit's message or contents).
Q4:
Already pushed to GitHub? → Yes → git revert <sha> creates an inverse commit — safe for shared history. Never reset a shared branch.
Q5:
Something is lost (deleted branch, accidental reset --hard)? → Yes → git reflog finds the dangling SHA, then git checkout <sha> or git branch recovered <sha>.
Reset 模式

三種 reset 模式：soft / mixed / hard

git reset 會把 HEAD 往回搬，但「工作區」和「暫存區」要不要跟著動？這就是三種模式的差別：

git reset moves HEAD backwards — but does the working dir and the index follow? That's what the three modes control:

模式 HEAD Index Working dir 用途
--soft移動不動不動把上一次 commit 拆回暫存區，重新合併成新 commit。"Uncommit" while keeping everything staged for a re-commit.
--mixed (default)移動重設不動commit 取消、變更退回工作區，最常用。Uncommit and unstage, but keep edits in working dir — the most common.
--hard ⚠移動重設重設（破壞性！）把所有東西回到指定 commit——你的本地修改會消失。Everything snaps to the target commit — local edits are gone.
🚨
--hard 是真的會刪檔！沒 commit 過的修改、沒 add 過的新檔案，全部會被毀掉，沒有 reflog 救不回來。下這個指令前先 git status 看一下工作區。 --hard really does delete files! Uncommitted edits and unstaged new files are gone forever — no reflog can save them. Always git status first.
Revert vs Reset

已經 push 了？只能用 revert

git reset 是「改寫歷史」——它把舊的 commit 從分支移除。如果你的同事已經 git pull 拉了那個 commit，再強推（push --force）會讓他們的 repo 出現詭異的衝突，大家都會討厭你。

git revert 則是「補一個反向 commit」：原本 commit 加 5 行，revert 就再 commit 一次把那 5 行刪掉。歷史完整保留，所有人 git pull 一下就同步了。

git reset rewrites history — it removes commits from the branch. If a collaborator has already pulled that commit, force-pushing creates a nasty divergence and everybody will hate you.

git revert instead adds a new "inverse commit": if the original added 5 lines, the revert commits a deletion of those 5 lines. History stays linear, and everyone catches up with a normal git pull.

# 安全：在共用分支撤銷一次 commit / Safe undo on a shared branch
$ git revert a1b2c3d
# 編輯器跳出，寫好 revert message → :wq
$ git push origin main

# 撤銷一段範圍（從 a1b2c3d 之後到 HEAD 全部反向）/ Revert a range
$ git revert a1b2c3d..HEAD
暫存改動

git stash：臨時抽屜

場景：你正在改 scRNA_clustering.R，改一半 PI 走進來說「上週那張 UMAP 圖能不能立刻給我？」——但你現在的工作區一團亂，又不想隨便 commit。git stash 就是這個救星：把目前所有未 commit 的變更收進一個暫存抽屜，工作區瞬間乾淨，可以安心切到別的 branch。

Scenario: you're halfway through scRNA_clustering.R when your PI walks in and asks for last week's UMAP plot right now. Your working dir is a mess and you don't want a junk commit. git stash is the rescue: it tucks every uncommitted change into a drawer, leaving your working dir clean to switch branches.

# 暫存當前所有變更 / Stash everything
$ git stash
$ git stash push -m "WIP: testing new clustering resolution"

# 連未追蹤的新檔案也一起收（很重要！）/ Include untracked files
$ git stash --include-untracked

# 看抽屜裡有什麼 / List stashes
$ git stash list
stash@{0}: On main: WIP: testing new clustering resolution
stash@{1}: On dev: half-done QC code

# 拿回最近一個（並從抽屜刪掉）/ Pop the latest stash
$ git stash pop

# 拿回但不刪（怕用錯的話）/ Apply without removing
$ git stash apply stash@{1}

# 確認不需要、丟掉 / Drop a stash
$ git stash drop stash@{0}
安全網

git reflog：你的 90 天安全網

Git 把每一次 HEAD 移動（commit、checkout、reset、merge、rebase）都記錄到 .git/logs/HEAD，預設保留 90 天。換句話說，就算你 reset --hard 把昨天的工作清乾淨，那個 commit 的 SHA 還在——只要還在 90 天內，git reflog 就能找到。

Every HEAD move — commit, checkout, reset, merge, rebase — is logged to .git/logs/HEAD for 90 days by default. So even if you reset --hard and wipe yesterday's work, the commit's SHA still exists. As long as you're inside the 90-day window, git reflog can find it.

# 看 HEAD 的歷史移動 / Show HEAD move history
$ git reflog
a1b2c3d (HEAD -> main) HEAD@{0}: reset: moving to HEAD~1
e4f5g6h HEAD@{1}: commit: add DESeq2 results table
9876543 HEAD@{2}: checkout: moving from dev to main
...

# 救回方法 1：直接 checkout 那個 SHA / Recover by SHA
$ git checkout e4f5g6h

# 救回方法 2：把它變成一個新 branch / Recover as a branch
$ git branch recovered-work e4f5g6h

# 救回方法 3：用 HEAD@{N} 語法 / Recover by reflog index
$ git reset --hard HEAD@{1}    # 回到 1 步前的狀態
🆘
記得這句話：「Got 99 problems? git reflog has the answer.」——任何「Git 弄丟東西」的災難，第一步永遠是 git reflog。 Repeat after me: "Got 99 problems? git reflog has the answer." Whenever Git seems to have lost something, the first command is always git reflog.

完整程式碼

撤銷指令對照範例

# 1. 撤銷工作區單一檔案（沒 add 過） / Revert one unstaged file
$ git restore differential_expression.R
# 等同舊版： git checkout -- differential_expression.R

# 2. 撤銷整個工作區所有未 add 的變更 / Revert ALL unstaged changes
$ git restore .

# 3. 從暫存區拿掉（unstage），但保留編輯 / Unstage but keep edits
$ git restore --staged config.yaml

# 4. 修正最後一次 commit 的訊息（沒 push 才能用）/ Fix last commit message
$ git commit --amend -m "fix: correct DESeq2 contrast formula"

# 5. 把剛漏掉的檔案補進最後一次 commit / Add a forgotten file to last commit
$ git add data/sample_sheet.csv
$ git commit --amend --no-edit
# --soft：commit 退回，但變更保留在暫存區 / Uncommit, keep staged
$ git reset --soft HEAD~1
# 用途：剛 commit 完發現訊息寫錯、或想拆成兩個 commit

# --mixed（預設）：commit 退回、unstage、編輯留在工作區 / Default
$ git reset HEAD~1
# 用途：把最近一個 commit 整個拆開，重新規劃 add 哪些

# --hard：什麼都消失，回到指定 commit / Destructive — everything wiped
$ git reset --hard HEAD~1     # ⚠ 用前先 git status

# 從 reflog 救回剛剛被 --hard 砍掉的東西 / Recover from reflog
$ git reflog
$ git reset --hard HEAD@{1}   # 回到 reset --hard 之前那一刻
# 撤銷一個已經 push 出去的 commit / Revert one pushed commit
$ git log --oneline -5
a1b2c3d # bad commit we want to undo
e4f5g6h # good

$ git revert a1b2c3d
# 編輯器跳出 → 寫 revert message → :wq
$ git push origin main

# 撤銷一段範圍（含起點到 HEAD）/ Revert a range
$ git revert a1b2c3d^..HEAD --no-edit

# 撤銷 merge commit（要指定 -m 1 主線父節點） / Revert a merge
$ git revert -m 1 <merge_sha>

# ❌ 千萬不要在共用分支用：
#   git reset --hard HEAD~1 && git push --force
#   會讓所有同事的 repo 爆炸
⚠️
強推紅線：共用分支（main、dev、release 分支）絕對不要 push --force。GitHub 可以設 branch protection 直接擋下，建議所有實驗室 repo 都開。 Force-push red line: never push --force on shared branches (main, dev, release branches). GitHub's branch protection blocks it — turn it on for every lab repo.
生資現場

生資實驗室常見撤銷場景

🧬 不小心 commit 了 BAM 大檔

還沒 push：git reset --soft HEAD~1 → 把 *.bam 加進 .gitignore → git restore --staged *.bam → 重新 commit。已 push 但想真正清掉 history：用 git filter-repo 或 BFG。

If unpushed: git reset --soft HEAD~1 → add *.bam to .gitignore → git restore --staged *.bam → recommit. If pushed and you must purge history, use git filter-repo or BFG.

🔑 API key 被 commit 進去

立刻在 GitHub 撤銷該 token（這比改 git 史更重要！），再用 git filter-repo --replace-text 重寫歷史，然後 force push。提醒所有人重新 clone。

First, revoke the token on GitHub — this matters more than rewriting Git history. Then git filter-repo --replace-text to scrub it, force push, and tell everyone to re-clone.

📊 改一半被 PI 叫去看別的 branch

git stash push -m "WIP clustering" → git switch last-week-plots → 看完之後 git switch - 回來 → git stash pop。乾淨無痕。

git stash push -m "WIP clustering" → git switch last-week-plots → when done, git switch - back → git stash pop. Clean and trace-free.

💥 reset --hard 把昨天工作清掉

深呼吸 → git reflog 找到昨天最後一次 commit 的 SHA → git branch recovered <sha>。只要還在 90 天內，工作就還在。

Breathe → git reflog to find yesterday's last commit SHA → git branch recovered <sha>. As long as you're within 90 days, the work is still there.
